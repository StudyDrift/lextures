## 1. Problem Statement

In standards-based courses, the meaningful signal is per-standard proficiency, not a
percentage. Mobile shows only points, so K-12 students (and parents, via M10.1) can't
see how they're doing against standards or read their report card on a phone.

## 2. Goals

- Per-course **mastery** list with proficiency level per concept, rendered as a compact,
  accessible heatmap (color + text label, never color-only).
- Student-facing **report cards** (read), with PDF download/share.
- Surface a **practice hint** for not-yet-assessed concepts.

## 3. Non-Goals

- Editing standards alignment/scales or generating report cards (instructor/admin/web).
- Deep evidence tracing (concept → contributing assignment list) — the backend's mastery
  heatmap view does not currently join to grading evidence; only a lightweight practice
  hint is shown instead. A future story can add this once the server exposes it.
- The org-configured SBG 1–5 scale UI (`/students/{id}/sbg/{period}`) — mobile uses the
  existing concept-mastery heatmap (0.0–1.0 score, same thresholds as web's
  `masteryLabel()`) as the single mastery signal, since it's already student-accessible
  and requires no new permission surface.

## 4. User Stories

- **As a student**, I open a course's Mastery tab and see which concepts I've mastered
  and which need work, with a clear label (not just a color).
- **As a student**, I open my report card on my phone and can save/share the PDF.

## 5. What already exists (reuse)

- **Server**: `GET /courses/{code}/enrollments/{enrollmentId}/mastery` (student- or
  staff-accessible per-concept mastery vector), `GET /report-cards/{cardId}/pdf`
  (student-accessible once they know the card id). **Mobile**: M6.1 grades patterns,
  M0.2 offline cache, M5.2 peer review as the template for a new self-contained feature
  module (API/models/logic/screens/navigation wiring).

## 6. Functional Requirements

- **FR-1.** A course's **Mastery** section MUST list concepts with the student's current
  proficiency level (Mastered / Developing / Beginning / At risk / Not yet assessed).
- **FR-2.** The list MUST be legible and accessible on a phone: color-coded plus a text
  label, never color-only.
- **FR-3.** A not-yet-assessed concept SHOWS a practice hint (points to the AI tutor).
  Deep evidence links are deferred (see Non-Goals).
- **FR-4.** A student-facing **report cards** list MUST be viewable (read-only, released
  cards only) with PDF download/share.
- **FR-5.** Proficiency thresholds MUST match web's `masteryLabel()` (≥0.8 mastered,
  ≥0.6 developing, ≥0.4 beginning, else at risk).

## 7. Non-Functional

- **Accessibility**: proficiency always has a text label, never color-only; heatmap is a
  navigable list (VoiceOver/TalkBack friendly), not an image.
- **Offline**: mastery list and report card list render from cache (M0.2).
- **Privacy**: report cards endpoint only returns the caller's own released cards.

## 8. Design / Approach

`Features/Mastery/` (iOS) / `features/mastery/` (Android): `CourseMasterySection` (course
workspace tab, compact concept list with proficiency), `ReportCardListView` /
`ReportCardListScreen` (More hub entry, released cards + PDF share sheet). New
`MasteryLogic` (proficiency thresholds, sorting, report-card filtering — unit tested on
both platforms). API in `LMSAPIMastery.swift` / `LmsApi.kt` +
`FileDownloadManager.fetchReportCardPdf`.

## 10. Acceptance Criteria

1. A course's Mastery tab lists concepts with proficiency labels matching web's
   thresholds. ✅
2. The list is legible and accessible (label + color, not color-only) on a phone. ✅
3. Not-yet-assessed concepts show a practice hint. ✅ (deep evidence links deferred, see
   Non-Goals)
4. Released report cards are viewable and the PDF can be downloaded/shared. Both apps
   build and their unit tests pass. ✅
