## Implementation notes (2026-07)

- **Entry**: Course workspace **Live** section when `liveSessionsEnabled`; Home dashboard **Live & upcoming** rail; deep links `/courses/{code}/live`.
- **API** (both platforms): `fetchCourseMeetings`, `fetchMeetingJoinInfo`, `patchMeeting`, `fetchMeetingAttendance`, `fetchCourseWhiteboards`, `fetchCourseWhiteboard` against existing virtual-meetings and whiteboard routes.
- **UI**: `Features/Live/` (iOS) and `features/live/` (Android) — meeting list grouped by live/upcoming/past, live banner, join + ical + detail sheet/dialog, staff start/end/cancel, attendance roster, read-only whiteboard viewer with pan/zoom.
- **Logic**: `LiveMeetingsLogic.swift` / `LiveMeetingsLogic.kt` — grouping, join eligibility, countdown, provider labels, planner calendar mapping, cross-course rail collection.
- **Attendance**: `GET /meetings/{id}/join` upserts attendance when the meeting is live (no separate client POST).
- **Calendar (M2.1)**: live meetings feed `PlannerCalendarEventKind.liveMeeting` via planner snapshot loaders.
- **Offline**: meeting lists cached per course (`OfflineCacheKey.liveMeetings`); join requires network.
- **Feature flag**: `ffMobileLiveMeetings` / `mobile_live_meetings` hides Live section and dashboard rail when off.
- **Whiteboard v1**: read-only native renderer (`WhiteboardRenderer`); staff can list/open boards; full authoring remains web (server whiteboard APIs are staff-gated).
- **Tests**: `LiveMeetingsLogicTests` (iOS), `LiveMeetingsLogicTest` + `WhiteboardElementTest` (Android unit).
- **i18n**: `mobile.live.*` in `clients/mobile/locales/*.json` (synced to iOS xcstrings and Android `strings.xml`).
