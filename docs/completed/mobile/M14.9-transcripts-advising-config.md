## Implementation notes (2026-07-14)

- **Access and rollout:** Profile entry and screens require `ffMobileAdminSettings` plus
  `global:app:rbac:manage`. Hub entry also requires at least one of `ffTranscripts` or
  `ffAdvisingIntegration`. Sub-views gate on their own feature flag; unauthorized views show
  the permission/flag fallback and do not call admin endpoints.
- **Hub:** `TranscriptsAdvisingAdminView` / `TranscriptsAdvisingAdminScreen` lists visible
  sections (Transcripts, Advising).
- **Transcripts:** load `GET /api/v1/admin/transcripts/config` and delivery failures from
  `GET /api/v1/admin/transcripts/requests`; save via `PUT /api/v1/admin/transcripts/config`.
  Webhook secret is write-only (placeholder when configured; only sent when changed).
- **Advising:** load/save `GET|POST /api/v1/admin/advising/config` ÔÇö appointment URL, degree
  audit provider (none/degreeworks/stellic), base URL, credentials ref, at-risk banner.
  Feeds student M7.8 advising surface.
- **States:** loading skeleton, save/saving/saved/error, configure-on-web link-out.
- **Logic & API:** `TranscriptsAdvisingAdminLogic` + `LMSFeatureModelsTranscriptsAdvisingAdmin` /
  `LmsFeatureModelsTranscriptsAdvisingAdmin.kt`; methods on `LMSAPIAdmin.swift` / `LmsApi.kt`.
- **Tests:** `TranscriptsAdvisingAdminLogicTests` (iOS) and `TranscriptsAdvisingAdminLogicTest`
  (Android) cover gating, section visibility, payload builders, URL validation, and decode.
  Both suites pass.
- **i18n:** 49 keys under `mobile.admin.transcriptsAdvising.*`, `mobile.admin.transcripts.*`,
  and `mobile.admin.advising.*` in `clients/mobile/locales`, synced to iOS catalog and Android
  `strings.xml`.

## 1. Problem Statement

Where enabled, admins configure **official transcripts** (formats/fields/release policy) and the
**advising** integration (notes/appointments config the student advising surface in M7.8 relies
on). Both are web-only, so an admin can't review or tweak them from a phone, even though students
already use advising on mobile.

## 2. Goals

- **Transcripts**: view/edit transcript configuration (fields, release policy) ÔÇö flag-gated.
- **Advising**: view/edit advising integration config ÔÇö flag-gated.
- Feed the student advising surface (M7.8) with correct config.

## 3. Non-Goals

- No generating/issuing individual transcripts here (that's a separate flow).
- No SIS credential entry (web/M14.8).

## 4. Personas & User Stories

- **As a registrar admin**, I review the transcript release policy from my phone.
- **As an advising admin**, I toggle an advising integration option on the go.

## 5. Functional Requirements

- **FR-1.** Each sub-view MUST require `rbac:manage` **and** its flag (`ffTranscripts` /
  `ffAdvisingIntegration`); absent otherwise.
- **FR-2.** MUST load and edit transcript config and persist.
- **FR-3.** MUST load and edit advising config and persist.
- **FR-4.** MUST show save/saving/saved/error states; require connectivity; audit edits.

## 6. Non-Functional Requirements

- **Performance** ÔÇö load/save < 700 ms p95.
- **Security** ÔÇö server-check permission + flag; audit edits.
- **Privacy & Compliance** ÔÇö transcripts + advising notes are sensitive FERPA data; config only,
  no student records displayed here; no PII in logs.
- **Accessibility** ÔÇö form controls labeled; Ôëą44 pt.
- **Internationalization** ÔÇö localized; RTL.

## 7. Acceptance Criteria

- **AC-1.** *Given* insufficient permission or a disabled flag, *Then* the sub-view is absent.
- **AC-2.** *Given* a transcript config edit saved, *Then* it persists and audits.
- **AC-3.** *Given* an advising config edit saved, *Then* it persists and M7.8 reflects it.
- **AC-4.** Both apps build.

## 8. Data Model

- No schema changes. Read/write config only; no student records cached.

## 9. API Surface

- Reuse transcripts + advising config endpoints (`/me/advising/config` and transcript settings
  endpoints). Verify shapes vs. the two web panels.

## 10. UI / UX

- Two cards (shown only when their flag is on): Transcripts config, Advising config ÔÇö each a
  simple form.
- States: fallback, loading, error, saved.
