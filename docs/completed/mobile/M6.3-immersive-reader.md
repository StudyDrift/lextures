## 1. Problem Statement

[M0.3](../completed/mobile/M0.3-accessibility.md) made the app work with VoiceOver/TalkBack
and Dynamic Type, but the **content-level** inclusive-reading tools the web ships are
absent: read-aloud of a page/assignment, captions/transcripts on course videos,
on-the-fly translation of content, and reader preferences (dyslexia-friendly font, line
spacing, theme). Students who rely on these — emergent readers, EL learners, dyslexic and
low-vision students — cannot consume content comfortably on mobile, which is exactly the
phone-first cohort.

## 2. Goals

- **Read-aloud** any rendered content surface (pages, assignment prompts, discussion
  posts, vibe activities) using on-device TTS with play/pause and follow-along highlight.
- **Captions/transcripts** on course video/audio: render VTT tracks, language picker,
  transcript view, and caption styling.
- **On-the-fly translation** of content into the viewer's preferred language via the
  existing course-translation endpoints.
- **Reading preferences**: font family (incl. dyslexia-friendly), size, spacing, and
  reading theme applied across content viewers, persisted via the server prefs where
  available.

## 3. Non-Goals

- No caption authoring/editing or translation authoring (web/admin).
- No new TTS/translation models — on-device TTS for read-aloud, server endpoints for
  translation/captions.
- Not changing the global app theme — these are reading-surface preferences.

## 4. Personas & User Stories

- **As an emergent reader**, I tap "Read aloud" and follow the highlighted text.
- **As an EL student**, I translate a page into my home language inline.
- **As a Deaf/HoH student**, I turn on captions and read the transcript of a course video.
- **As a dyslexic student**, I switch to a dyslexia-friendly font with more line spacing.

## 5. Functional Requirements

- **FR-1.** Rendered content surfaces MUST expose a **read-aloud** control using on-device
  TTS (`AVSpeechSynthesizer` / Android `TextToSpeech`) with play/pause/stop, rate
  control, and word/sentence follow-along highlight where feasible.
- **FR-2.** Video/audio players MUST render **caption tracks** from the server VTT
  (`/files/{object_id}/captions/{caption_id}/vtt`), offer a track/language picker, a
  toggle, and a scrollable **transcript** view; caption styling MUST respect reading prefs.
- **FR-3.** A content surface MUST offer **translate** into the viewer's preferred locale
  via course-translation endpoints, showing translated text with a "show original" toggle
  and provenance ("machine translated").
- **FR-4.** A **reading-preferences** panel MUST let the user set font (incl.
  dyslexia-friendly), size, line/letter spacing, and reading theme; preferences MUST
  apply across content viewers and persist (server `reading-preferences` when available,
  else local).
- **FR-5.** All tools MUST be reachable from a single, consistent **reader toolbar** on
  content surfaces and MUST be keyboard/switch/VoiceOver operable.
- **FR-6.** Tools MUST degrade gracefully: no captions track → hide caption UI; no
  translation for the locale → notice; TTS unavailable → disable with explanation.

## 6. Non-Functional Requirements

- **Performance** — read-aloud starts < 500 ms; translation request < 1 s p95 on warm
  network; caption rendering keeps A/V sync.
- **Security/Privacy** — on-device TTS keeps text local; translation goes to the existing
  server endpoint only; no content text sent to third parties from the client.
- **Accessibility** — this *is* the accessibility story; all controls labeled, ≥44pt,
  Dynamic Type, reduced-motion (no flashing highlight), polite announcements; meets WCAG
  2.1 AA for the added UI.
- **Reliability** — translation failure falls back to original with a notice; caption
  fetch failure hides the toggle, never blocks playback.
- **Internationalization** — picker lists server-supported locales; RTL for translated
  RTL languages.
- **Offline** — last-fetched translation/captions cached for offline reading (M0.2);
  TTS works offline.

## 7. Acceptance Criteria

- **AC-1.** *Given* a content page, *When* read-aloud starts, *Then* TTS reads it with a
  visible follow-along highlight and pause/stop work.
- **AC-2.** *Given* a course video with a caption track, *When* captions are enabled,
  *Then* synced captions render, the language picker switches tracks, and a transcript is
  available.
- **AC-3.** *Given* content and a preferred locale, *When* the user taps translate, *Then*
  translated text shows with a "show original" toggle and "machine translated" label.
- **AC-4.** *Given* reading preferences set to a dyslexia font + larger spacing, *When*
  applied, *Then* all content viewers reflect it and it persists across sessions.
- **AC-5.** *Given* a surface with no captions/translation available, *Then* those tools
  hide/disable with a clear reason.
- **AC-6.** Both apps build; axe-equivalent + screen-reader scripts pass on the new UI.

## 8. Data Model

- No server schema changes. Client persists reading preferences locally (mirroring the
  server `reading-preferences` resource when present); caches caption VTT + translated
  content per item for offline.

## 9. API Surface

- Reuse caption endpoints (`/files/{object_id}/captions`, `…/{caption_id}/vtt`),
  course-translation (`course-translation-api.ts` / `course_translation.go`,
  `translation.go`), and `reading-preferences-api.ts`. Verify shapes when implementing.
  No new endpoints.

## 10. UI / UX

- A consistent **reader toolbar** on content viewers: Read-aloud · Translate · Aa
  (reading prefs) · (CC on media).
- Read-aloud: floating mini-player (play/pause/stop, rate), follow-along highlight.
- Captions: standard player overlay + transcript sheet + language menu.
- Translate: inline replace with "show original" + provenance chip.
- Reading prefs: bottom sheet with font/size/spacing/theme, live preview.
- States: tool unavailable (hidden/disabled), translation loading, offline-cached notice.

## 11. AI / ML Considerations

- Translation/caption generation are server-side (existing); mobile consumes outputs.
  Read-aloud is on-device TTS (no model hosting). Surface "machine translated"/"auto
  captions" provenance per the server's labeling; PII stays on device for TTS.

## 12. Integration Points

- **iOS:** new `Features/Reader/ReaderToolbar.swift`, `ReadAloudController.swift`

- Coordinates with [M3.6](./M3.6-library-ereserves-oer.md) (media captions on library

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Follow-along highlight hard across rich markdown | M | M | Sentence-level highlight first; word-level where simple |
| Translation latency/quality on long content | M | M | Translate visible blocks first; cache; provenance label |
| Reading prefs leak into non-content UI | L | M | Scope prefs to content viewers only |

## 15. Rollout Plan

- Flag: `mobile_immersive_reader`. Internal → a11y pilot → GA.
- Sequence: reading prefs + read-aloud → captions → translation.
- Rollback: flag hides the reader toolbar.

## 16. Test Plan

- **Unit** — TTS controller state; prefs persistence; VTT parsing/sync.
- **Integration** — translation round-trip + fallback; caption track switching.
- **E2E** — read a page aloud; caption a video; translate + revert; apply dyslexia font.
- **Accessibility** — screen-reader operation of every control; reduced-motion highlight;
  contrast of caption styles.
