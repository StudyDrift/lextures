## 1. Problem Statement

The library e-reserves and OER catalog give students access to required readings,
library-licensed resources, and openly licensed materials. On the web these appear both
as a browsable catalog and as course-attached library resources (with access logging for
licensing/usage reporting). On mobile there is nothing: a student assigned a library
reading can't open it without a browser, and OER discovery is unavailable.

## 2. Goals

- Browse/search the **library catalog** and **OER** from the app and open results.
- Show a course's **e-reserve / library resources** within the course workspace and open
  them (link-out or in-app preview as the resource allows).
- Record **access/link events** so usage/licensing reporting stays accurate.
- Respect access controls and licensing terms exactly as the server enforces them.

## 3. Non-Goals

- No library admin/config (`/admin/library/config`) — web-only.
- No instructor curation of e-reserves on mobile v1 (attach happens on web).
- Not a full e-book reader — open licensed content via its provided viewer/link.

## 4. Personas & User Stories

- **As a student**, I open my course's e-reserves and read an assigned chapter on my phone.
- **As a self-learner**, I search OER and open an openly licensed resource.
- **As a librarian/instructor**, I trust that mobile opens count toward usage reporting.

## 5. Functional Requirements

- **FR-1.** The course workspace MUST gain a **Library** section (when the course has
  library resources) listing e-reserves/library resources with title, type, and access
  state, opening each resource.
- **FR-2.** A **Library/OER browse** surface (in the More hub) MUST search
  `library/search` and `oer/search`, render results, and open them.
- **FR-3.** Opening a library resource MUST record an **access/link event**
  (`library-resources/{id}/access`, `library-link-events`) before/at open so usage
  reporting is accurate.
- **FR-4.** Access-controlled or licensed resources MUST honor server gating (paywall,
  seat, or licensing notices) and show the server-provided access affordance rather than
  forcing an open.
- **FR-5.** Resources that need a provider/web context MUST open via in-app web view or
  external link with a "best on a larger screen" affordance, not a broken native attempt.

## 6. Non-Functional Requirements

- **Performance** — search results < 500 ms p95; catalog list paginates.
- **Security** — access events authenticated; licensing/DRM honored; no caching of
  licensed content beyond what terms allow.
- **Privacy & Compliance** — usage events are the existing licensing requirement; no
  extra tracking; FERPA-safe.
- **Accessibility** — list + reader controls labeled, ≥44pt, Dynamic Type, captions where
  the resource is media (coordinate with [M6.3](./M6.3-immersive-reader.md)).
- **Reliability** — access event failure surfaces but doesn't block reading when the
  resource itself is reachable; retry queued (M0.2).
- **Internationalization** — result metadata respects locale; RTL.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course with e-reserves, *When* the student opens the Library
  section, *Then* resources list and open, and an access event is recorded server-side.
- **AC-2.** *Given* a search query, *When* run against library/OER, *Then* results render
  and open; no results shows an empty state.
- **AC-3.** *Given* a licensed/gated resource, *When* opened, *Then* the server's access
  notice/affordance shows instead of a broken open.
- **AC-4.** *Given* a resource best viewed on web, *When* opened, *Then* an in-app web
  view / external open with a clear affordance is used.
- **AC-5.** Both apps build.

## 8. Data Model

- No server changes. Client caches recent library searches and last-opened resources;
  queues access events offline for replay (M0.2).

## 9. API Surface

- Reuse `GET /api/v1/library/search`, `GET /api/v1/oer/search`,
  `GET /api/v1/courses/{c}/library-resources/{item_id}`,
  `POST /api/v1/courses/{c}/library-resources/{item_id}/access`,
  `POST /api/v1/courses/{c}/library-link-events`,
  `…/structure/modules/{module_id}/library-resources`. Verify shapes vs `library-api.ts`,
  `oer-api.ts`, and `he_library_http.go`.

## 10. UI / UX

- Course workspace **Library** chip → resource list → resource detail/open.
- More-hub **Library/OER** → search + browse → result → open (web view or external).
- States: empty (no resources / no results), loading skeleton, gated/licensed notice,
  error banner, offline (cached list + notice).
- Reuse file/preview affordances from [M3.2](../completed/mobile/M3.2-course-files.md).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| DRM/licensing blocks native rendering | H | M | Web view / provider link with clear affordance |
| Missed access events skew usage reports | M | M | Record event at open; queue + retry offline |
| OER vs library result-shape divergence | M | M | Thin per-source adapter; verify at build |

## 15. Rollout Plan

- Flag: `ffMobileLibraryEreserves` on `PlatformFeatures` (defaults on; set `false` to hide Library chip + hub entry and fall back to web-only module routing).
- Rollback: flag hides the Library chip + hub entry.

## 16. Test Plan

- **E2E** — open e-reserve → event recorded; OER search → open; offline replay.
- **Accessibility** — list/reader labels; captions hand-off.
