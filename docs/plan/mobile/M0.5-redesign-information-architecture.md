## 1. Problem Statement

The app shipped a clean 5-tab shell sized for a handful of read-only surfaces. We are
now adding live classes, a real-time course feed, evaluations, library/e-reserves,
immersive reading, instructor analytics, advising, vibe activities, and universal
search. There is no room for these in a flat tab bar, no role adaptation (a parent and
an instructor see the same student-shaped shell), and course detail is a four-chip row
that will not hold ten surfaces. Without an information-architecture redesign, every new
feature gets bolted on as an orphaned screen with no discoverable entry point, and the
"do everything the web does" goal collapses into a maze.

## 2. Goals

- Establish a **role-aware shell** that presents the right primary destinations to
  students, instructors, parents, and self-learners without forking the codebase.
- Introduce a scalable **course workspace** sub-navigation that can hold every
  course-scoped surface (modules, grades, discussions, feed, live, people, files,
  evaluations, library, more) without nesting users into dead ends.
- Add a persistent, predictable place for **search** and **secondary destinations**
  ("More" hub) so new features are always one obvious tap away.
- Keep the existing brand system, dark mode, accessibility, and offline affordances —
  this is a re-organization, not a re-skin.
- Define the IA contract every later mobile story plugs into (where does my surface
  live, how is it reached, how does it adapt by role).

## 3. Non-Goals

- No new colors/fonts/brand changes — reuse `LexturesTheme`/`LexturesType`.
- No new backend, no new business features — those land in their own stories.
- Not building the destination screens themselves (live, feed, etc.) — only their
  navigation slots, role gating, and empty entry points.
- No web redesign; web IA is unchanged.

## 4. Personas & User Stories

- **As a student**, I want a bottom bar focused on my day (Home, Courses, Inbox,
  Notebooks, Profile) and a course screen that exposes everything that course offers
  without hunting.
- **As an instructor**, I want my shell to foreground teaching tools (a Teach/grading
  surface, attendance, insights) instead of pretending I'm a student.
- **As a parent**, I want a shell scoped to my children, not a student's course list.
- **As a self-learner**, I want catalog/enroll and my paths within easy reach.
- **As any user**, I want one search affordance that finds courses, content, people,
  and actions from anywhere.

## 5. Functional Requirements

- **FR-1.** The shell MUST select a **role-aware tab set** from the viewer's effective
  roles (student / instructor-TA / parent / self-learner), defaulting to student when
  ambiguous, and MUST let a multi-role user switch active context (e.g. "Teaching" vs
  "Learning") without re-login.
- **FR-2.** The bottom bar MUST remain ≤5 primary destinations; overflow destinations
  MUST live in a **"More" hub** (a grid/list surface reachable from Profile or a 5th
  tab) that lists every secondary feature with icon, label, and role gating.
- **FR-3.** Course detail MUST become a **course workspace** with a scalable, scrollable
  segmented sub-nav whose sections are **driven by what the course actually enables**
  (feature flags / presence of data / viewer role), not a hardcoded four. Sections at
  full scope: Overview, Modules, Grades, Discussions, Feed, Live, People, Files,
  Attendance, Evaluations, Library, More.
- **FR-4.** Every shell header MUST expose a **search entry point** (icon → [M0.6](./M0.6-universal-search.md))
  and a notifications/bell with unread state.
- **FR-5.** Navigation MUST be **deep-link addressable** (cooperating with
  [M0.1](../completed/mobile/M0.1-push-deep-links.md)): any destination reachable by tap
  MUST be reachable by a route, so push/notifications/search can jump straight in.
- **FR-6.** Section availability MUST degrade gracefully: a course that doesn't enable
  a surface MUST hide its chip (never show a chip that opens an empty/"not available"
  screen).
- **FR-7.** Tab/section state MUST survive process death and tab switches (preserve the
  existing per-tab `NavigationStack` / `rememberSaveable` pattern).
- **FR-8.** The redesign MUST keep all currently shipping destinations reachable (no
  feature regressions); a parity checklist MUST be verified before flag flip.

## 6. Non-Functional Requirements

- **Performance** — tab/section switch < 150 ms; course workspace chips render from
  cached structure without a blocking fetch; no layout jank on chip overflow scroll.
- **Security** — role/context derived from server-authoritative enrollment roles; never
  trust a cached role to unlock instructor surfaces (re-check on entry).
- **Privacy & Compliance** — parent context MUST scope to authorized children only;
  no cross-tenant leakage when switching context.
- **Accessibility** — bottom bar and sub-nav are VoiceOver/TalkBack ordered, ≥44pt
  targets, the selected section is announced, Dynamic Type reflows chips; honor
  reduced-motion for tab transitions.
- **Scalability** — adding a future surface MUST be a registry entry, not a shell
  rewrite (define a `MobileDestination` registry per platform).
- **Reliability** — unknown/disabled destinations fail closed (hidden), never crash.
- **Observability** — emit a nav event (`destination`, `role_context`, `entry_method`)
  for funnel analysis; respect existing `X-Platform`/`X-App-Version` headers.
- **Internationalization** — all labels externalized; sub-nav reflows under RTL.
- **Backward compatibility** — old deep links/entry points keep working; ship behind a
  flag with a one-release fallback to the current shell.

## 7. Acceptance Criteria

- **AC-1.** *Given* an instructor-only enrollment, *When* they open the app, *Then* the
  bottom bar foregrounds teaching destinations and a "Learning"/"Teaching" context
  switch is available; a student-only account never sees teaching tabs.
- **AC-2.** *Given* a course that enables discussions, feed, and live but not a library,
  *When* the student opens it, *Then* the workspace shows exactly those chips and hides
  Library; tapping any chip opens a real (possibly empty-state) surface, never a 404.
- **AC-3.** *Given* any root tab, *When* the user taps the header search icon, *Then*
  universal search opens (M0.6) and returns to the originating tab on dismiss.
- **AC-4.** *Given* a push deep link to a course feed, *When* tapped, *Then* the app
  routes directly to that course's Feed section with the back stack intact.
- **AC-5.** *Given* a multi-role parent+student, *When* they switch to parent context,
  *Then* the shell scopes to their children and no student course list leaks.
- **AC-6.** *Given* the feature flag off, *When* the app launches, *Then* the prior
  5-tab shell renders unchanged (clean rollback).
- **AC-7.** Both apps build; a parity checklist confirms every pre-redesign destination
  is still reachable.

## 8. Data Model

- **No server schema changes.** Client-only: a `MobileDestination` registry (id, title,
  icon, required roles, course-feature predicate, route) per platform; a persisted
  "active role context" preference in the existing secure/prefs store.

## 9. API Surface

- **No new endpoints.** Reuses `GET /api/v1/me` (identity), course enrollment roles
  (existing `course-view-as`/enrollment role source), and per-course structure/feature
  flags already fetched for course detail to decide which workspace chips to show.

## 10. UI / UX

- **Shell.** Keep the floating pill bottom bar. Role contexts:
  - *Student / Self-learner*: Home · Courses · Notebooks · Inbox · Profile (current).
  - *Instructor/TA*: Home · Courses · **Teach** (grading backlog + attendance + insights
    hub) · Inbox · Profile.
  - *Parent*: Home(children) · Children · Calendar · Inbox · Profile.
  - 5th item / Profile doubles as the **"More" hub** when overflow exists.
- **"More" hub.** A searchable grid of secondary destinations (Calendar, Live, Library,
  Catalog, Paths, Reading, Portfolio, Credentials, Settings, Advising…), each gated by
  role/flag, with an empty state when none apply.
- **Course workspace.** Gradient hero retained; a horizontally scrolling segmented chip
  row (reuse `LMSSegmentedChips`/`LmsSegmentedChips`) whose items come from the
  destination registry filtered by course feature flags + role. Overflow chips scroll;
  a trailing "More" chip opens a sheet listing the rest.
- **Context switch.** A header control (avatar long-press or a small segmented control)
  toggles Learning/Teaching for multi-role users.
- **States.** Each new slot ships an empty/loading/error/offline state (the destination
  stories fill content; this story guarantees the slot + a graceful placeholder).
- **Accessibility.** Tab order documented; selected chip announced; reflow under Dynamic
  Type and RTL verified.
- **Copy & i18n.** New labels (Teach, Feed, Live, Library, People, Evaluations, More,
  Learning/Teaching) externalized.

## 13. Dependencies & Sequencing

- Must ship **before** the new-surface stories (M3.5, M3.6, M6.3, M7.5–M7.8, M11.3) so
  they have IA slots; those stories register their destination + add content.
- Must ship **with/after** [M0.6](./M0.6-universal-search.md) (header search target) —
  can land the search entry icon dark and wire it when M0.6 arrives.
- Shared infra: existing theme, nav stacks, deep-link router. No new infra.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| IA churn breaks existing entry points | M | H | Parity checklist + keep flag-gated old shell for one release |
| Role detection wrong → instructor tools shown to students | L | H | Server-authoritative role re-check on entry; fail closed |
| Chip overflow unreadable on small screens / large type | M | M | Scrollable row + "More" sheet; Dynamic Type tests |
| Multi-role context confusion | M | M | Explicit, persistent context switch; clear labels |

## 15. Rollout Plan

- Flag: `mobile_ia_redesign` (default off → internal → pilot → GA).
- Sequence: ship registry + role-aware shell behind flag → migrate course workspace →
  add More hub → wire search/bell → parity verify → flip flag.
- Dogfood with internal multi-role accounts (student+instructor+parent).
- GA when parity checklist passes on both platforms and no P1 nav defects.
- Rollback: flip flag → prior 5-tab shell.

## 16. Test Plan

- **Unit** — destination registry filtering by role × course-feature matrix; context
  switch state machine.
- **Integration** — role-derived tab set from `/me` + enrollment roles; workspace chips
  from course structure/flags.
- **End-to-end** — per persona: open app → reach every applicable surface; deep link →
  correct section; context switch scopes correctly.
- **Accessibility** — VoiceOver/TalkBack traversal of bar + chips; Dynamic Type + RTL
  reflow; reduced-motion.
- **Manual** — parity checklist (every pre-redesign destination reachable); flag-off
  regression.
