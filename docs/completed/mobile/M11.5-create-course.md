## Implementation notes (2026-07-09)

- **Logic & API**: `CourseCreateLogic` (iOS/Android) ports web starter templates + validation; client methods `createCourse`, `patchCourseSyllabus`, `createCourseModule`, `fetchOrgTerms` on `LMSAPI` / `LmsApi`.
- **Wizard**: `CourseCreateView` / `CourseCreateScreen` — Basics → Syllabus template → First module (traditional) or competency hand-off; back re-enters Basics with `PUT` (no duplicate); cancel after Step 1 confirms and keeps the course.
- **Entry**: **New course** `+` on Courses list (iOS toolbar / Android header), gated by `ffMobileCreateCourse` + `global:app:course:create` + online.
- **Feature flag**: `ffMobileCreateCourse` on `PlatformFeatures` / `MobilePlatformFeatures` (default **off**).
- **Competency v1**: course is created as `competency_based`; full competency/assessment builder deferred (Step 3 hand-off opens the course).
- **i18n**: `mobile.createCourse.*` keys in `clients/mobile/locales`; synced to Localizable.xcstrings / Android strings.
- **Tests**: `CourseCreateLogicTests` (iOS) and `CourseCreateLogicTest` (Android) — permission gate, validation, template parity, update-vs-create, syllabus payload.
- **xcodeproj**: regenerated via `scripts/generate_xcodeproj.py`.

## 1. Problem Statement

The web app lets a permitted user stand up a new course in a guided **3-step wizard**: **Basics**
(title, description, traditional-vs-competency mode, term, grade level → `POST /api/v1/courses`),
**Syllabus template** (pick a starter scaffold or blank → `PATCH .../syllabus`), and **First
module** (traditional: optional `POST .../modules`) **or Competencies** (competency-based: each
competency becomes a module with outcomes, sub-outcomes, and linked assessments). On mobile there is
**no way to create a course** — the Courses list has no "New course" action and the API client has no
create methods. A teacher who wants to spin up a class, a sandbox, or a self-paced course from a
phone simply can't, which undercuts the "manage your teaching from mobile" story that M11/M13 set up.

## 2. Goals

- A permission-gated **New course** entry on the mobile Courses surface that opens a create flow.
- A **Basics** step at full parity: title (required), description, **course mode**
  (traditional / competency-based), **term** (from org terms), **grade level** → creates the course
  via `POST /api/v1/courses`.
- A **Syllabus template** step: choose a starter scaffold (K-12, Higher ed, Self-paced, Bootcamp,
  Onboarding) or **blank**, applied via `PATCH .../syllabus`.
- A **finish** step that mirrors web: traditional ⇒ optional first module, then open the course.
- Land the user **inside the new course** on finish (parity with web's navigate-to-course), ready for
  M13.1 settings and authoring.
- Feel native and forgiving: clear per-step validation, a resumable draft (the course exists after
  step 1), and a safe exit that doesn't strand a half-built course.

## 3. Non-Goals

- **Not** a redesign of course authoring — modules, assignments, quizzes, and outcomes keep their own
  surfaces/stories; this wizard only creates the *shell* and (traditional) an optional first module.
- **Not** the full on-device **competency builder** in v1 (the web competency path chains ~6 writes
  per competency: module + outcome + sub-outcomes + assessment + outcome-link). v1 lets the user
  **choose** competency mode (course is typed `competency_based`); the in-app competency+assessment
  builder is a fast-follow (§18) or added if scope allows.
- **No** course copy / import (Canvas import, "create from course") — those are separate (web
  `CourseCatalogImportMenu`; mobile follow-up).
- **No** offline creation — course creation requires connectivity (multi-step server writes); the flow
  is online-only, not queued via M0.2.
- **No** hero image / theme / scheduling in the wizard — those live in **M13.1 course settings**,
  reachable immediately after finish.

## 4. Personas & User Stories

- **As an instructor**, I want to create a new course from my phone so I can start a class the moment
  I decide to, not when I'm next at a laptop.
- **As a K-12 teacher**, I want a K-12 starter syllabus pre-filled so a new section isn't a blank page.
- **As a self-learner/creator**, I want to spin up a self-paced course quickly and open it to add
  content.
- **As a department admin**, I want course creation gated to permitted staff so learners can't create
  courses.

## 5. Functional Requirements

- **FR-1.** The **New course** entry MUST appear on the Courses surface **only** when the cached
  permissions grant `global:app:course:create`; otherwise it is absent (server re-checks on write).
- **FR-2.** **Step 1 (Basics)** MUST require a non-empty title, allow description, and let the user
  choose **course mode** (traditional / competency-based), **term** (org terms list), and **grade
  level**. On continue it MUST call `POST /api/v1/courses` and hold the returned `courseCode`.
- **FR-3.** Re-entering Step 1 after creation MUST update the existing course (parity with web's
  `putCourse` on back-then-continue), not create a duplicate.
- **FR-4.** **Step 2 (Syllabus template)** MUST offer the same starter set as web plus **Blank**, and
  applying a non-blank template MUST `PATCH .../syllabus` with that template's sections.
- **FR-5.** **Step 3 (traditional)** MUST allow an optional first-module title (defaulted from the
  template's suggested title) and, if provided, `POST .../modules`; a **Skip** option MUST exist.
- **FR-6.** On finish the app MUST refresh permissions/roles (the creator is enrolled as teacher) and
  **navigate into the new course** (course workspace / M13.1 settings entry).
- **FR-7.** Every step MUST show inline validation, a submitting state, and a non-blocking error on
  failure; failures MUST NOT advance the step.
- **FR-8.** If the user exits after Step 1, the created course MUST remain valid and openable from the
  Courses list (no orphaned/broken shell); the app SHOULD confirm before abandoning an in-progress
  wizard.
- **FR-9.** Progress MUST be shown as "Step N of 3" with labeled stops (Basics · Syllabus · Module/
  Competencies), matching web.
- **FR-10.** Competency mode: v1 MUST at minimum create the course typed `competency_based` and open
  it; the full competency/assessment builder MAY ship in v1 or as the fast-follow (§18), behind the
  same flow.

## 6. Non-Functional Requirements

- **Performance** — each step's write is a single request; p95 < 700 ms. Term list fetched once and
  cached. No blocking spinner longer than the request.
- **Security** — client gates on cached permission for affordance only; the server authorizes every
  write on `global:app:course:create` and scopes the course to the user's org. No client trust.
- **Privacy & Compliance** — no PII beyond course metadata the user types; FERPA-neutral.
- **Accessibility** — WCAG 2.1 AA / VoiceOver + TalkBack: labeled fields, error text tied to inputs,
  ≥44 pt targets, focus moves to the first invalid field, step changes announced, dynamic type.
- **Scalability** — bounded, single-user interactive flow; no batch concerns for the traditional path.
- **Reliability** — resumable after Step 1 (course persisted); each write is independent so a mid-flow
  failure never corrupts the course. Guard against double-submit.
- **Observability** — emit `course.created` (mode, template, has-first-module) analytics; log write
  failures per step to spot where users drop.
- **Internationalization** — all copy via mobile i18n keys; RTL-safe; template names/summaries
  localized; grade-level and term labels localized.
- **Backward compatibility** — additive client feature behind a flag; no schema or contract change.

## 7. Acceptance Criteria

- **AC-1.** *Given* a user with `global:app:course:create`, *When* they open Courses, *Then* a **New
  course** action is present; *Given* a user without it, *Then* it is absent.
- **AC-2.** *Given* Step 1 with a title, *When* I continue, *Then* `POST /courses` runs and I advance
  to Syllabus holding the new `courseCode`; *Given* an empty title, *Then* I see a validation error
  and stay.
- **AC-3.** *Given* I go back to Basics and change the title, *When* I continue, *Then* the existing
  course is updated (no duplicate course is created).
- **AC-4.** *Given* Step 2, *When* I pick a starter template and continue, *Then* the syllabus is
  patched with that template; *When* I pick Blank, *Then* no syllabus write occurs.
- **AC-5.** *Given* traditional mode Step 3, *When* I enter a module title and finish, *Then* a module
  is created and the app opens the new course; *When* I Skip, *Then* no module is created and the
  course still opens.
- **AC-6.** *Given* any step write fails, *Then* an error shows, I remain on that step, and no partial
  advance occurs.
- **AC-7.** *Given* I exit after Step 1, *Then* the course exists in my Courses list and opens
  normally.
- **AC-8.** *Given* competency mode, *When* I finish, *Then* the course is created as
  `competency_based` and opens (plus, if in scope, its competencies/modules exist).
- **AC-9.** Both iOS and Android build; permission-gating and step logic are unit-tested.

## 8. Data Model

- **No schema changes.** All writes go through existing endpoints and tables (`course.courses`,
  syllabus, `course.modules`, and — competency phase — outcomes/sub-outcomes/assessments/outcome
  links). The creator is auto-enrolled as teacher server-side (existing `CreateCourse` behavior).
- Client holds transient wizard draft state (title, description, mode, termId, gradeLevel, selected
  template, first-module title, and — competency phase — the competency drafts) plus the created
  `CoursePublic` once Step 1 succeeds.
- **Starter templates** are client constants on web (`course-create-templates.ts`); mobile MUST port
  the same definitions into a shared logic module (below) — there is no template endpoint.

## 9. API Surface

- **Reuse (no backend work):**
  - `POST /api/v1/courses` — body `{ title, description, courseType?, termId?, gradeLevel? }` → returns
    `CoursePublic`.
  - `PATCH /api/v1/courses/{code}/syllabus` — `{ sections, requireSyllabusAcceptance }`.
  - `POST /api/v1/courses/{code}/modules` — `{ title }`.
  - Org terms: existing `fetchOrgTerms(orgId)` equivalent.
  - Competency phase: `POST .../outcomes`, `PATCH .../outcomes/{id}`, `POST
    .../outcomes/{id}/sub-outcomes`, `POST .../modules/{id}/assignments` | `.../quizzes`, `POST
    .../outcomes/{id}/links` (mobile already has the **outcomes/sub-outcomes** methods in
    `LMSAPIOutcomes.swift`; module/assignment/quiz create methods are **new client** work).
- **New mobile API-client methods** (client-only): `createCourse`, `patchCourseSyllabus`,
  `createCourseModule` on both platforms; competency-phase adds `createModuleAssignment`,
  `createModuleQuiz`, `addCourseOutcomeLink`.
- No rate-limit change; document nothing new server-side (endpoints already in OpenAPI).

## 10. UI / UX

Mirror the web wizard's structure, tuned for one-hand phone use.

**Entry point.** Add a **New course** affordance to the mobile Courses list (toolbar `+` on iOS /
FAB or top-app-bar action on Android), visible only when permitted. Tapping opens the wizard modally
(full-screen cover / dedicated route).

**Step 1 — Basics.** Title (required), Description (multiline), **Course mode** segmented control
(Traditional / Competency-based) with a one-line explainer, **Term** picker (org terms; "None"
allowed), **Grade level** picker. Primary button: *Continue* → creates/updates the course.

**Step 2 — Syllabus template.** A vertical list of template cards (name + summary) plus **Blank**,
single-select, matching the web starter set. *Continue* applies the template (or nothing for Blank).

**Step 3 — Finish.**
- *Traditional:* one field, **First module title** (pre-filled from the template's suggestion),
  with **Skip** and **Create & open** actions.
- *Competency (if in v1 scope):* a repeatable **Competency** builder (title + description, each with
  sub-outcomes and an assessment kind/title), mirroring web validation; otherwise Step 3 is a short
  "Competencies are added on the course page" hand-off that still finishes and opens the course.

**Progress & nav.** "Step N of 3" header with labeled stops (Basics · Syllabus · Module/Competencies);
Back preserves entered values; a confirm-on-cancel guards accidental exit after Step 1.

**States.**
- *Loading:* term list skeleton on Step 1; per-step submitting spinner on the primary button.
- *Empty:* no terms → term picker shows "None / Ongoing".
- *Error:* inline banner per step; writes never advance on failure; retry-in-place.
- *Success:* finish → brief confirmation, then push into the new course.
- *Offline:* the entry is disabled with "Connect to create a course" (online-only).

**Accessibility.** Field labels + error association, focus to first invalid field, ≥44 pt targets,
announce step transitions, dynamic type, RTL. Segmented control and pickers fully VoiceOver/TalkBack
operable.

**Copy & i18n keys** (new, `mobile.createCourse.*`): `title`, `step.basics|syllabus|module`,
`field.title|description|mode|term|gradeLevel`, `mode.traditional|competency`, `template.blank`,
per-template `name`/`summary`, `firstModule.label|skip`, `action.continue|createOpen`,
`error.titleRequired|createFailed|syllabusFailed|moduleFailed`, `cancel.confirm`.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Half-finished wizard leaves a bare course | M | L | Course is valid after Step 1; it appears in Courses and opens; confirm-on-cancel; finish is optional beyond Step 1 |
| Competency path is heavy for phone (many chained writes) | M | M | Phase it: v1 creates typed course + opens; full builder as fast-follow; each write independent + validated |
| Duplicate course on back→continue | M | M | Track created `courseCode`; re-enter updates via PUT (parity with web), never re-POST |
| Template drift between web and mobile | L | M | Port templates into a shared, unit-tested constant; add a parity test/checklist |
| Permission mismatch (affordance shown, write 403s) | L | M | Gate on cached permission for UX; server is source of truth; surface 403 cleanly |
| Connectivity drop mid-flow | M | M | Online-only with clear messaging; per-step retry-in-place; no partial advance |

## 15. Rollout Plan

- **Feature flag:** `ffMobileCreateCourse` on `PlatformFeatures` / `MobilePlatformFeatures`, default
  **off**. (Competency builder MAY sit behind a sub-flag `ffMobileCreateCourseCompetency` if phased.)
- **Sequencing:** ship API-client + traditional flow behind the flag → internal dogfood → instructor
  pilot → GA; enable competency builder when it lands.
- **GA criteria:** AC-1…AC-9 green on both platforms; create success rate healthy in dogfood; no rise
  in orphaned/duplicate courses.
- **Rollback:** flip the flag off — the New course entry disappears; already-created courses are
  unaffected.

## 16. Test Plan

- **Unit** (`CourseCreateLogicTests` / `CourseCreateLogicTest`): title validation; back→continue
  updates vs. creates; template selection → syllabus payload; first-module default from template;
  competency validation (if in scope); permission-gate predicate.
- **Integration:** `POST /courses` happy path + 403 without permission; syllabus patch; module create;
  term fetch/cache.
- **End-to-end:** permitted user creates traditional course with template + first module → lands in
  course; blank + skip path; competency mode creates typed course + opens; back-navigation preserves
  fields and avoids duplicates; non-permitted user sees no entry.
- **Security:** server enforces `global:app:course:create`; org-scoping; no client-only trust.
- **Accessibility:** VoiceOver/TalkBack pass on all steps; focus-to-first-error; dynamic type; RTL.
- **Performance:** step writes within p95 target; term-list caching.
- **Manual exploratory:** connectivity loss mid-flow; rapid double-tap on Continue/Finish; cancel
  after Step 1; both course modes; each starter template.

## 18. Open Questions

1. **Competency builder scope** — full on-device competency+assessment builder in v1, or ship
   traditional-first and add the builder as a fast-follow (recommended)?
2. **Entry placement** — toolbar `+` vs. FAB vs. an item in the Courses overflow menu; align with the
   M0.5 IA conventions.
3. **Exit semantics** — keep the Step-1 course on cancel (recommended, matches web) or offer "discard
   this draft course" (would need a delete/rollback)?
4. **Template parity** — port all five web starters, or a curated mobile subset + Blank for v1?
5. **Course-copy / import** — is a mobile "create from existing course" or Canvas import wanted later
   (separate story)?
6. **First-module step** — keep the optional first-module step, or drop straight into the course and
   let authoring happen there?
