## 1. Problem Statement

Course evaluations open in a window near term end and are a required, time-sensitive
task; the same machinery powers mid-term surveys/polls. On the web students complete them
in a few taps. On mobile there's no way to even see an open window, so phone-first
students simply don't respond — directly lowering evaluation response rates that
institutions care about.

## 2. Goals

- Show a student when an **evaluation/survey window is open** for a course (and as a
  to-do/notification), with deadline.
- Let the student **complete and submit** the evaluation (Likert scales, single/multi
  select, free text) on a phone.
- Honor **anonymity** and one-submission rules exactly as the server enforces them.
- Let instructors **view results** (read-only, after release) where permitted.

## 3. Non-Goals

- No evaluation **template/window authoring** (admin/web — `evaluation-templates`,
  `evaluation-windows`).
- No analytics beyond rendering released results.
- Not the same as graded quizzes (these are ungraded feedback instruments).

## 4. Personas & User Stories

- **As a student**, I get a reminder my course evaluation is open and complete it on my
  phone before it closes.
- **As a student**, I trust my responses are anonymous when the evaluation says so.
- **As an instructor**, after release I read my evaluation results on mobile.

## 5. Functional Requirements

- **FR-1.** The app MUST surface open evaluation/survey windows: a **to-do**/notification
  and a course workspace **Evaluations** entry, each showing the deadline and submission
  state (`GET /courses/{c}/evaluations/status`).
- **FR-2.** The form MUST render every question type the window defines (Likert/rating,
  single/multi choice, free text, optional) and validate required items before submit.
- **FR-3.** Submit MUST POST to `…/evaluations/{window_id}/submit`, be **idempotent**
  (one submission), and confirm success; after submit the window shows "submitted."
- **FR-4.** **Anonymity** MUST be respected and clearly communicated (the client must not
  attach identity beyond what the endpoint requires).
- **FR-5.** Instructors/authorized roles MUST be able to view **released results**
  (`…/evaluations/results`) read-only; unreleased results are hidden with a notice.
- **FR-6.** Closed/already-submitted windows MUST show a clear state and block re-submit.

## 6. Non-Functional Requirements

- **Performance** — form loads < 500 ms; submit confirms < 1 s p95.
- **Security** — authenticated submit; results gated by role + release; no identity leak
  on anonymous evals.
- **Privacy & Compliance** — anonymity guarantees honored; no response content in logs;
  FERPA-safe results visibility.
- **Accessibility** — Likert/rating controls labeled and operable by screen reader,
  ≥44pt, Dynamic Type; error association on required items.
- **Reliability** — idempotent submit (guard double-tap / retries); partial progress
  preserved locally until submit.
- **Internationalization** — question text/locale honored; RTL.
- **Offline** — answers fill offline and submit on reconnect (M0.2), once.

## 7. Acceptance Criteria

- **AC-1.** *Given* an open window, *When* the student opens the app, *Then* a to-do +
  workspace entry show it with the deadline.
- **AC-2.** *Given* the form, *When* completed and submitted, *Then* the server records
  one submission, the window shows "submitted," and re-submit is blocked.
- **AC-3.** *Given* an anonymous evaluation, *Then* no client-attached identity beyond the
  endpoint contract, and the UI states anonymity.
- **AC-4.** *Given* released results and an authorized instructor, *Then* results render
  read-only; unreleased shows a notice.
- **AC-5.** *Given* a required item left blank, *Then* submit is blocked with a clear error.
- **AC-6.** Both apps build.

## 8. Data Model

- No server schema changes. Client holds transient form state + a local draft until
  submit; submit guarded by an idempotency key.

## 9. API Surface

- Reuse `GET /api/v1/courses/{c}/evaluations/status`,
  `POST /api/v1/courses/{c}/evaluations/{window_id}/submit`,
  `GET /api/v1/courses/{c}/evaluations/results`. Verify shapes vs
  `course-evaluations-api.ts` and `course_evaluations.go`.

## 10. UI / UX

- To-do/notification → Evaluation form (sectioned: rating grids, choices, free text) →
  submit → confirmation.
- Workspace **Evaluations** chip (when a window exists or results are viewable).
- Anonymity banner; deadline countdown.
- Results (instructor): summary distributions + comments, read-only.
- States: not-open, open (deadline), submitted, closed, unreleased-results, error, offline.

## 11. AI / ML Considerations

None (unless server provides comment summarization — render only, defer).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Double submission on retry | M | H | Idempotency key + server one-submission rule |
| Anonymity expectations vs client metadata | L | H | Send only endpoint-required fields; audit payload |
| Rating-grid a11y on small screens | M | M | Vertical per-item layout; screen-reader test |
