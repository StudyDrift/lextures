## Implementation notes (2026-07-05)

- **Entry**: More hub **Advising** tile gated on `ffAdvisingIntegration` (server) and `ffMobileAdvising` (client rollout flag, default on). Deep link `/advising` unchanged.
- **API** (both platforms): `fetchAdvisingNotes`, `fetchDegreeProgress`, `fetchMyAdvisingConfig` against existing student advising endpoints. Models in `LMSFeatureModelsAdvising.swift` / `LmsFeatureModelsAdvising.kt`; API in `LMSAPIAdvising.swift` / `LmsApi.kt`.
- **UI**: `Features/Advising/AdvisingView.swift` (iOS) and `features/advising/AdvisingScreen.kt` (Android) ‚Äî advisor card (derived from notes), at-risk hold banner (`degreeProgress.atRisk`), degree-audit summary, notes timeline (newest first), and **Schedule advising appointment** button opening the configured external `appointmentUrl` (web parity; no native advising slot API exists today).
- **Logic**: `AdvisingLogic.swift` / `AdvisingLogic.kt` ‚Äî feature gating, `visibleToStudent` defense-in-depth filter, note sorting, appointment URL resolution, offline booking guard.
- **Offline**: notes and degree progress cached (`OfflineCacheKey.advisingNotes` / `degreeProgress`); staleness chip when replaying aged cache; booking disabled offline with explanation.
- **Open questions resolved**: (1) Web uses external scheduler URL, not conference/office-hours slots ‚Äî mobile matches. (2) Holds come from `GET /me/degree-progress` `atRisk` when banner enabled, not a separate endpoint.
- **Calendar (M2.1)**: External appointments are not in-app calendar events; native slot booking can be added when a backend slot source exists.
- **Tests**: `AdvisingLogicTests` (iOS) and `AdvisingLogicTest` (Android) ‚Äî gating, note visibility, appointment URL, offline booking guard.
- **i18n**: `mobile.advising.*` in all five `clients/mobile/locales/*.json` (synced).
- **Xcode**: `clients/ios/scripts/patch_xcodeproj_advising.py`.
