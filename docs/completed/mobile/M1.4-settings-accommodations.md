## 1. Problem Statement

The current profile is thin. Students need to edit their profile, control app behavior
(theme, language, biometric, notifications, dyslexia/read-aloud), and — importantly —
see and use their **approved accommodations** (extra time, etc.), which must be visible
and honored on mobile too.

## 2. Goals

- Edit **profile** (name, avatar, contact, pronouns where applicable).
- **Preferences**: theme, language override (M0.4), biometric (M1.2), notifications
  (M2.2), accessibility toggles (M0.3).
- View **my accommodations** (approved supports) and confirm they're applied (e.g.,
  extended quiz time reflected in M4.1).

## 3. Non-Goals

- Approving/granting accommodations (staff/admin/web). Account deletion flows beyond
  linking to the privacy center.

## 4. User Stories

- **As a student**, I update my display name and avatar.
- **As a student**, I switch to dark mode and Spanish.
- **As a student with extended time**, I confirm my accommodation is active and see the
  extra time in quizzes.

## 5. What already exists (reuse)

- **Mobile**: `ProfileView`/`ProfileTab`, notifications list, theme; M1.2/M2.2/M0.3/M0.4
  toggles. **Server (no changes)**: profile/settings + accommodations endpoints; the
  engine (12.10) already enforces accommodations server-side.

## 6. Functional Requirements

- **FR-1.** A **Profile** screen MUST allow editing name, avatar, and permitted contact
  fields, persisted server-side.
- **FR-2.** A **Settings** screen MUST expose: theme (system/light/dark), **language**
  override (M0.4), **biometric lock** (M1.2), **notification preferences** (M2.2), and
  **accessibility** toggles (dyslexia font, read-aloud defaults — M0.3), plus links to
  privacy center / trust center / about / sign out.
- **FR-3.** A **My Accommodations** view MUST list the student's approved accommodations
  with plain-language descriptions and where each applies.
- **FR-4.** Accommodations affecting timed activities (e.g., extended time) MUST be
  reflected in the relevant flow (M4.1 reads the server-applied limit — verify it shows
  the accommodated time).
- **FR-5.** Settings MUST persist across devices where server-backed; device-only prefs
  stored locally.

## 7. Non-Functional

- **Privacy**: accommodations are sensitive — display only to the student; respect minor
  policy. **Accessibility**: the settings screen itself is a model of M0.3 compliance.
  **Offline**: read cached; edits queue.

## 8. Design / Approach

Extend `Features/Profile/`: `SettingsView` (sectioned toggles + links), `EditProfileView`,
`MyAccommodationsView`. Wire toggles to M1.2/M2.2/M0.3/M0.4. Calls in `LMSAPIFeatures.swift`/
`LmsApi.kt`.

## 12. Implementation notes (completed)

**Approach.** The existing `ProfileView` (iOS) / `ProfileTab` (Android) already served as
the settings surface, so rather than introduce a parallel, duplicate `SettingsView` we
extended that screen with the missing pieces and added the two genuinely new sub-screens.
This keeps a single settings surface and avoids redundant code.

**iOS** (`clients/ios`):
- `Core/Design/ThemePreference.swift` *(new)* — observable system/light/dark override,
  persisted in `UserDefaults`, applied at the app root via `preferredColorScheme`.
- `Features/Profile/EditProfileView.swift` *(new)* — edits first/last name, avatar URL,
  phone via `GET`/`PATCH /api/v1/settings/account`; refreshes the shell so the new name
  shows everywhere.
- `Features/Profile/MyAccommodationsView.swift` *(new)* — lists active supports from
  `GET /api/v1/me/accommodations` in plain language with the course scope and date window.
- `Features/Profile/ProfileSettingsCards.swift` *(new)* — appearance picker, personal
  (edit profile + accommodations) entries, and privacy/trust/accessibility links.
- `Core/LMS/AccountAccommodationModels.swift` *(new)*, `Core/LMS/LMSAPIFeatures.swift`,
  `Core/Design/ProfileAvatarView.swift` (avatar with image/initials fallback), strings in
  all five `Localizable.xcstrings` locales, `project.pbxproj` (regenerated).
- Tests: `LexturesTests/SettingsAccommodationsTests.swift`.

**Android** (`clients/android`):
- `core/design/ThemePreference.kt` *(new)* — process-singleton Compose state honored by
  `isDarkTheme()` / `LexturesTheme`.
- `features/profile/EditProfileScreen.kt`, `features/profile/MyAccommodationsScreen.kt`
  *(new)*; `features/profile/ProfileTab.kt` gains the appearance picker, edit-profile /
  accommodations entries, and privacy/trust links; `core/lms/LmsApi.kt` +
  `core/lms/LmsFeatureModels.kt` gain the account/accommodation calls; strings added to
  `values`, `values-es`, `values-fr`, `values-ar`, `values-en-rXA`.
- Tests: `core/lms/SettingsAccommodationsTest.kt`.

**Backend:** unchanged (endpoints `GET`/`PATCH /api/v1/settings/account` and
`GET /api/v1/me/accommodations` already existed).

**AC-4 (extended-time in quizzes).** The accommodations engine (12.10) applies extended
time server-side; mobile renders the server-provided `timeLimitMinutes`. The interactive
quiz timer (M4.1) is not yet on mobile, so the accommodated value flows through the moment
that timer consumes the server limit. My Accommodations now surfaces "Extended time on
timed activities" so students can confirm the support is active.

**Verification:** iOS `xcodebuild test` (all suites) and Android `testDebugUnitTest`,
`assembleDebug`, and `lintDebug` all pass; swiftlint reports no errors.
