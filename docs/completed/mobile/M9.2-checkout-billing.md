## Policy decision (resolved)

**v1 uses option (2) — compliant web-checkout handoff** (Safari / Chrome Custom Tabs), not in-app StoreKit/Play Billing. Paid digital courses complete on Stripe-hosted checkout in the system browser; entitlements are validated server-side on return. Path bundle purchases without a course checkout API continue to use the M9.1 explore-page web handoff.

## Implementation notes (2026-07-02)

- **API layer**: `LMSAPIBilling.swift` / `LmsApi.kt` — `POST /api/v1/checkout` or `/api/v1/billing/checkout`, `POST /api/v1/checkout/quote`, `GET /api/v1/me/entitlements`, `GET /api/v1/me/transactions`, `GET /api/v1/billing/portal`, `GET /api/v1/internal/entitlements/check`. Models in `LMSFeatureModelsBilling.swift` / `LmsFeatureModelsBilling.kt`. `BillingLogic` for URLs, labels, and feature gating (`ffStripeBilling`, `ffPaymentsEnabled`, `ffTaxCollection`).
- **Purchase flow**: `PurchaseFlowSheet` / `PurchaseFlowSheet` — tax quote when enabled, then opens Stripe checkout URL with `checkout/success?course_id=` and `checkout/cancel` return URLs.
- **Billing screen**: Profile → **Billing** when billing features are on — entitlements, transactions, **Manage subscription** (Stripe Customer Portal in browser).
- **Return handling**: `CheckoutReturnOverlay` polls entitlement check (server source of truth), then routes into the purchased course workspace. Deep links: `/checkout/success`, `/checkout/cancel`, `/me/billing` wired in `DeepLinkRouter` (+ Android manifest path prefixes).
- **Catalog integration**: `CourseLandingView` / `CourseLandingScreen` show **Purchase** instead of generic web handoff when billing is enabled.
- **Tests**: `BillingLogicTests` (iOS), `BillingLogicTest` + extended `DeepLinkRouterTest` (Android).
