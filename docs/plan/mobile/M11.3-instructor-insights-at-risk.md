## 1. Problem Statement

Instructors monitor class health on the web: an at-risk list, "what's working" engagement
insights, and per-student progress. On mobile, an instructor can take attendance and post
an announcement but has zero visibility into who is falling behind or how the class is
doing. The most valuable mobile moment for a teacher — a quick check between classes —
isn't supported, so intervention waits until they're back at a computer.

## 2. Goals

- A **class health snapshot** per course: at-risk count, engagement/insight highlights,
  ungraded backlog (link to [M6.1](../completed/mobile/M6.1-grades-feedback.md)/grading).
- An **at-risk students** list with the risk reason and a path to act (message the
  student, view progress).
- A **per-student progress** view (grades, submissions, activity) read-only.
- **"What's working"** engagement insights summarized for a phone.

## 3. Non-Goals

- No full reporting/export, custom report building, or CSV (web/admin —
  `reports`, `ai-reports`, `outcomes-report`).
- No gradebook grid editing (web).
- No changing risk model/config — render server outputs only.

## 4. Personas & User Stories

- **As an instructor**, between classes I open a course and see how many students are
  at-risk and why.
- **As an instructor**, I tap an at-risk student and message them or view their progress.
- **As an instructor**, I glance at "what's working" to see engagement trends.

## 5. Functional Requirements

- **FR-1.** A course **Insights** surface (Teach context / workspace chip for staff) MUST
  show a health snapshot: at-risk count, engagement highlights, ungraded backlog total
  (linking to grading), and recent activity.
- **FR-2.** An **at-risk list** MUST render students with risk indicators/reasons
  (from the at-risk endpoint), sorted by severity, each opening a student detail.
- **FR-3.** **Student detail** MUST show that student's grades summary, submission status,
  and activity (read-only), with a **message student** action (reuse Inbox compose).
- **FR-4.** A **"what's working"** view MUST summarize engagement insights (top/struggling
  items, participation) in a phone-readable form.
- **FR-5.** All surfaces MUST be **role-gated** to staff with appropriate scope and MUST
  show empty/loading/error/offline states.
- **FR-6.** Mobile MUST link out to web for the full reports/export with a "best on a
  larger screen" affordance rather than reimplementing them.

## 6. Non-Functional Requirements

- **Performance** — snapshot < 600 ms p95; lists paginate; charts are lightweight.
- **Security** — staff-only, scope-checked server-side on every fetch; never trust cached
  role; student PII handled per FERPA.
- **Privacy & Compliance** — FERPA: only authorized roster within the instructor's scope;
  no risk data in logs; no cross-section leakage.
- **Accessibility** — data summarized non-visually (don't rely on color alone for risk);
  charts have text equivalents; ≥44pt; Dynamic Type.
- **Reliability** — partial endpoint failure degrades that card only.
- **Internationalization** — numbers/dates localized; RTL.
- **Offline** — last snapshot cached with staleness indicator; read-only offline.

## 7. Acceptance Criteria

- **AC-1.** *Given* a staff member in a course, *When* Insights opens, *Then* the snapshot
  shows at-risk count, engagement highlights, and ungraded backlog linking to grading.
- **AC-2.** *Given* the at-risk list, *When* a student is tapped, *Then* their read-only
  progress shows and "message student" opens a prefilled compose.
- **AC-3.** *Given* a non-staff user, *Then* Insights is not reachable.
- **AC-4.** *Given* "what's working," *Then* engagement trends render with text
  equivalents (not color-only).
- **AC-5.** *Given* offline, *Then* the cached snapshot shows with a staleness notice.
- **AC-6.** Both apps build.

## 8. Data Model

- No server schema changes. Client caches the latest snapshot + at-risk list per course
  for offline read.

## 9. API Surface

- Reuse instructor-insights, at-risk, student-progress, and what's-working endpoints
  (`instructor_insights.go`, `at_risk.go`, `student-progress-api.ts`,
  `instructor-insights-api.ts`). Verify exact routes/shapes vs
  `course-at-risk.tsx`/`course-whats-working.tsx` when implementing.

## 10. UI / UX

- **Teach** context (M0.5) hub or staff **Insights** workspace chip → snapshot cards →
  at-risk list → student detail.
- Risk shown with icon + label + reason (not color alone).
- "What's working" as compact insight cards with text summaries.
- "View full reports on web" link-out.
- States: empty (no risk/insufficient data), loading skeleton, error per card, offline.

## 11. AI / ML Considerations

- At-risk scoring and some insights are model-derived server-side; mobile renders outputs
  with provenance/confidence where the server provides it. No client model; no PII to
  third parties. Show "predicted" framing, never present risk as deterministic fact.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Presenting predicted risk as fact | M | H | "Predicted"/confidence framing; reason shown; link to evidence |
| FERPA scope leakage | L | H | Server scope-checks every fetch; no client widening |
| Charts heavy / inaccessible | M | M | Lightweight + text equivalents; no color-only encoding |

## 15. Rollout Plan

- Flag: `mobile_instructor_insights`. Internal → instructor pilot → GA.
- Sequence: snapshot → at-risk list → student detail → what's-working.
- Rollback: flag hides Insights.

## 16. Test Plan

- **Unit** — risk sort/format; snapshot aggregation; role gating.
- **Integration** — each endpoint render; student detail compose prefill.
- **E2E** — staff opens Insights → at-risk → message student; non-staff blocked; offline.
- **Security** — scope checks; PII handling; staff-only.

## 17. Documentation & Training

- Note predictive framing and where to go for full reports.
