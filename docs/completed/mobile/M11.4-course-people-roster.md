> Let a teacher view the roster of their course on mobile — students, TAs, and

## 1. Problem Statement

A teacher opening the "People" chip on a course today sees a generic "not available on
mobile" placeholder (`CourseDestinationPlaceholder`), even though the tab is already
promoted into the course workspace nav for staff. Web's People page
(`course-enrollments.tsx`, reached via `/courses/{code}/enrollments`) lets a teacher see
who's enrolled (role, section, last access, invitation/state), add/remove people, and
message a student. On mobile, a teacher who wants to check roster size, confirm a
student is enrolled before class, or message a parent-invited student mid-class has to
switch to a browser — the exact "no laptop" failure this program exists to close.

## 2. Goals

- Let a teacher **view the full course roster** (students, TAs, co-teachers) with role,
  section, and last-access, from the phone.
- Let a teacher **search/filter** the roster (by name, role, section) for large classes.
- Let a teacher **message a person** directly from their roster row (reuses the
  enrollment message endpoint — no new inbox thread flow to build).
- Let a teacher **remove an enrollment** (e.g. a duplicate or mistaken add) for parity
  with the one destructive action teachers reach for most on web.
- Ship both iOS and Android to parity, replacing the placeholder.

## 3. Non-Goals

- **Adding people by email / invite flow** (web bulk-add UI) — stays web-only for now;
  mobile is read + light-action, not roster administration.
- **Editing a person's role or section, incomplete-grade management, group
  assignment** — web-only (`onPatchEnrollment`, groups panel); complex forms better
  suited to a larger screen.
- **Approving/declining pending invitations** — web-only for this story.
- **Progress/mastery drill-down per student** — covered by M11.3 (instructor
  insights/at-risk), not duplicated here.

## 4. User Stories

- **As a teacher**, I open my course's People tab and see everyone enrolled, grouped
  by role, so I know who's in my class.
- **As a teacher**, I search the roster for a student's name before answering a
  question about their enrollment.
- **As a teacher**, I tap a student and send them a quick message without leaving the
  course.
- **As a teacher**, I remove a mistakenly-added enrollment from my phone.
- **As a TA/co-teacher**, I see the same roster (read) as the primary teacher.

## 5. What already exists (reuse)

- **Server (no changes)**: `GET /api/v1/courses/{course_code}/enrollments` (roster
  list — role, section, state, last access), `DELETE
  .../enrollments/{enrollment_id}`, `POST .../enrollments/{enrollment_id}/message`
  (`server/internal/httpserver/course_enrollments_http.go`,
  `courses_routes.go:216-236`), `GET /api/v1/courses/{course_code}/sections` (section
  filter options).
- **Mobile shell**: `CourseWorkspaceSection.people` case already exists with label
  (`mobile.ia.course.people`), deep-link segment (`"people"`), and gating
  (`course.viewerIsStaff && course.isSectionsEnabled`) in
  `Core/Routing/MobileDestinations.swift` (iOS) /
  `core/navigation/MobileDestinations.kt` (Android); `DeepLinkRouter` already maps
  `people`/`enrollments` segments to `CourseDeepLinkSection.People`. Only the content
  view is missing — `CourseDetailView.swift:203` / `CourseDetailScreen.kt:440` route
  `.people` to `CourseDestinationPlaceholder`.
- **Reference pattern**: `TakeAttendanceView.swift`/`Screen.kt` for a staff-only
  roster UI built on `fetchCourseSections`; `CourseSection` model
  (`LMSFeatureModelsMobile.swift`) for section filter chips.
- **Web model to mirror**: `CourseEnrollment` type in `course-enrollments.tsx:58-74`
  (id, userId, displayName, avatarUrl, role, roleDisplay, lastCourseAccessAt, section
  fields, state, invitationPending).

## 6. Functional Requirements

- **FR-1.** The People tab MUST list all course enrollments (students, TAs,
  co-teachers) with display name, avatar, role, and section.
- **FR-2.** The list MUST be searchable by name and filterable by role and section.
- **FR-3.** A teacher MUST be able to tap a person to see enrollment detail (role,
  section, last access, enrollment state) and a **Message** action.
- **FR-4.** The Message action MUST send via `POST
  .../enrollments/{enrollment_id}/message` with subject + body.
- **FR-5.** A teacher with remove rights MUST be able to remove an enrollment, with a
  confirmation step before the destructive `DELETE` call.
- **FR-6.** Only staff (teacher/TA/co-teacher) MAY see the People tab; students MUST
  NOT see it (already enforced by the existing `viewerIsStaff` gate — server also
  enforces via auth on the list/message/delete endpoints).
- **FR-7.** Pending/invited enrollments MUST show an "invited" badge (read-only; no
  approve/decline on mobile).

## 7. Non-Functional

- **Performance**: roster list paginates or virtualizes for classes with 100+
  enrollments; search/filter is client-side against the loaded page for typical class
  sizes, matching web's approach.
- **Offline**: roster is a read path — shows last-cached list with a staleness
  indicator per cross-cutting conventions (M0.2); remove/message are online-only
  actions (destructive/messaging actions are not queued offline).
- **Accessibility**: roster rows and role/section filter chips are VoiceOver/TalkBack
  labelled; remove action requires an explicit confirmation dialog, not a swipe-only
  gesture.
- **Security**: remove and message actions rely on existing server-side authz
  (same checks web's `onRemoveEnrollment`/message flow use) — no new client-side
  permission logic invented.

## 8. Design / Approach

Build `CoursePeopleView` (iOS) / `CoursePeopleScreen` (Android) as the real content
for `CourseWorkspaceSection.people`, replacing the placeholder branch in
`CourseDetailView.swift:203` / `CourseDetailScreen.kt:440-442`. Structure:

1. Fetch roster via a new `fetchCourseEnrollments(courseCode:)` call (mirrors
   `fetchCourseSections` pattern in `LMSAPIMobileWave.swift` /
   Android equivalent) hitting the existing `GET .../enrollments` endpoint.
2. Group by role (Teachers/TAs first, then Students) with a search field and
   role/section filter chips (reuse `CourseSection` fetch already used by
   `TakeAttendanceView`).
3. Row tap → detail sheet with Message (opens a lightweight subject/body compose
   calling the message endpoint directly — not the full Inbox thread UI) and, for
   authorized staff, Remove (confirm dialog → `DELETE`).
4. Empty/loading/error states per cross-cutting conventions.

## 10. Acceptance Criteria

1. A staff member opens a course, taps People, and sees the full roster grouped by
   role with section labels — no more placeholder.
2. Search narrows the list by name; role/section filter chips work.
3. Tapping a person opens detail with Message; sending calls the enrollment message
   endpoint and confirms success.
4. Remove requires confirmation, calls `DELETE`, and updates the list on success;
   errors surface a message (matches web's error handling).
5. Students do not see the People tab (gate unchanged); TAs/co-teachers see the same
   read roster as the primary teacher.
6. Invited-but-pending enrollments show an "invited" badge, read-only.
7. Offline: last-loaded roster shows with a staleness indicator; remove/message are
   disabled offline with a clear reason. Both apps build.

## 11. Phasing

1. Roster fetch + list UI (grouped, search, section/role filters) replacing the
   placeholder on both platforms.
2. Row detail sheet + Message action.
3. Remove action with confirmation; offline staleness indicator; Android parity;
   verification.
