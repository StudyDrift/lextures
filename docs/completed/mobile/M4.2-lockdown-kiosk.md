## Implementation notes (2026-07)

- **Detection**: `QuizLogic.requiresDeviceLockdown` gates kiosk mode (`lockdownMode === 'kiosk'`) and optional proctoring `required` flag; `needsLockdownConsent` covers `one_at_a_time` and `kiosk` (web parity).
- **Pre-start consent**: `QuizIntroView` / `QuizIntroScreen` show an explicit lockdown explanation modal before starting secured delivery.
- **LockdownController** (iOS + Android):
  - **iOS**: Guided Access guidance, idle-timer disable, screenshot + screen-recording detection, guided-access exit detection → focus-loss events with student warning banner.
  - **Android**: `FLAG_SECURE`, `startLockTask()` with pinning guidance when lock task is unavailable, lifecycle focus-loss reporting.
- **API**: `fetchQuizProctoringConfig` added on both platforms (best-effort; returns null when unavailable).
- **Release**: lockdown deactivates on quiz taker disappear / submit.
- **i18n**: `mobile.quiz.lockdown.*` keys in all `clients/mobile/locales/*.json` (synced).
- **Tests**: `QuizLogicTests` / `QuizLogicTest` cover kiosk and device-lockdown helpers.

## 10. Acceptance Criteria

1. ✅ A secured quiz enters lockdown with explicit consent and a clear explanation.
2. ✅ App-switch/exit attempts are recorded via focus-loss and warned; screenshots blocked
   where the platform allows.
3. ✅ Lockdown releases cleanly on submit. Non-secured quizzes are unaffected. Both apps build.

## 11. Phasing

1. ✅ Secured detection + pre-start explanation + enter/exit lock (iOS Guided Access / Android pinning).
2. ✅ FLAG_SECURE / screenshot-recording detection + focus-loss reporting.
3. ✅ Violation policy + clean release; Android parity; verification.
