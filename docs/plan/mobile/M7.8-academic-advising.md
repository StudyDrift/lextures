## 1. Problem Statement

In higher ed, an advisor is assigned to a student, shares advising notes/holds, and meets
them on a schedule. The web exposes the student's advising notes and an advising
appointment scheduler. On mobile a student can't see their advisor, read shared notes,
or book an advising session — so a core HE support workflow is desktop-locked.

## 2. Goals

- Show the student **who their advisor is** and any **shared advising notes**.
- Let the student **book / view advising appointments** via the existing scheduler.
- Surface advising **holds/alerts** the student needs to act on.
- Keep it lean — reuse the [M7.3](./M7.3-office-hours.md) booking and
  [M10.2](./M10.2-conference-booking.md) slot patterns rather than a parallel stack.

## 3. Non-Goals

- No advisor-side caseload management/note authoring on mobile (web).
- No degree-audit/planning tool (separate, out of scope here).
- No admin advising config (`/admin/advising/*`).

## 4. Personas & User Stories

- **As a student**, I see my assigned advisor and the notes they've shared with me.
- **As a student**, I book an advising appointment from my phone and add it to my calendar.
- **As a student**, I see an advising hold and what I must do to clear it.

## 5. Functional Requirements

- **FR-1.** An **Advising** entry (More hub / profile) MUST show the student's advisor(s)
  and student-visible advising notes (`GET /me/advising-notes`), newest first.
- **FR-2.** The app MUST show **advising config/availability** (`GET /me/advising/config`)
  and let the student **book a slot** via the existing scheduler/booking endpoint, with
  an "add to calendar" ical option.
- **FR-3.** Advising **holds/alerts** (where present) MUST be shown with the required
  action and link/affordance to resolve.
- **FR-4.** Notes flagged staff-only MUST never be shown (render only student-visible
  notes the server returns).
- **FR-5.** Booked/upcoming advising appointments MUST appear in
  [M2.1](../completed/mobile/M2.1-calendar-todos.md).

## 6. Non-Functional Requirements

- **Security** — only student-visible notes returned by the server are shown; booking
  authenticated; no advisor-private content reachable.
- **Privacy & Compliance** — FERPA: advising notes are sensitive; no caching beyond
  session needs; no notes in logs.
- **Accessibility** — notes/booking labeled, ≥44pt, Dynamic Type.
- **Reliability** — booking idempotent; double-book guarded by the slot endpoint.
- **Internationalization** — times in user tz/locale; RTL.
- **Offline** — last-fetched notes readable with a staleness indicator; booking requires
  connectivity.

## 7. Acceptance Criteria

- **AC-1.** *Given* an assigned advisor, *When* Advising opens, *Then* the advisor and
  student-visible notes show; staff-only notes never appear.
- **AC-2.** *Given* availability, *When* the student books a slot, *Then* it's recorded,
  confirmed, and offered as a calendar add; it appears in the in-app calendar.
- **AC-3.** *Given* an advising hold, *Then* it shows with the required action.
- **AC-4.** *Given* offline, *Then* cached notes show with a staleness notice; booking is
  disabled with an explanation.
- **AC-5.** Both apps build.

## 8. Data Model

- No server schema changes. Client caches advisor + visible notes for offline read; no
  local persistence of booking beyond confirmation.

## 9. API Surface

- Reuse `GET /api/v1/me/advising-notes`, `GET /api/v1/me/advising/config`, the conference/
  scheduler **slot booking** endpoint (`/conference-slots/{slotId}/book`, `…/ical`) or the
  advising-specific scheduler route. Verify shapes vs `advising-api.ts`, `scheduler-api.ts`,
  and `advising_http.go`.

## 10. UI / UX

- More-hub **Advising** → advisor card + notes list + holds banner + "Book appointment."
- Booking reuses the [M7.3](./M7.3-office-hours.md)/[M10.2](./M10.2-conference-booking.md)
  slot picker.
- States: no advisor assigned, empty notes, holds present, offline (cached + notice), error.
