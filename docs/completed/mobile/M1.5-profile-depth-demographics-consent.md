## 1. Problem Statement

Orgs collect self-reported demographics, custom profile fields, and research-study
consent on the web. A phone-first user can never complete these, leaving required
profile data blank and locking them out of opt-in research and any flow that gates on a
complete profile. These are short forms that belong in the mobile profile.

## 2. Goals

- Let a user view and edit their **demographics** (self-reported, optional) on mobile.
- Render and submit **org-defined custom profile fields** (typed: text/select/date/bool).
- Surface available **research studies** with informed-consent text and a clear
  opt-in/opt-out, honoring withdrawal.
- Keep all of it privacy-first: clearly labeled optional, with rationale.

## 3. Non-Goals

- No admin authoring of custom fields or studies (web-only).
- No analytics on the demographic values beyond what the server already does.
- Not the accommodations engine (that's [M1.4](../completed/mobile/M1.4-settings-accommodations.md)).

## 4. Personas & User Stories

- **As a student**, I complete required custom profile fields my school added.
- **As a student**, I update optional demographics or choose "prefer not to say."
- **As a research participant**, I read the consent, opt in, and can withdraw later.

## 5. Functional Requirements

- **FR-1.** Profile MUST gain a **"Personal details"** section listing demographics and
  custom fields, each marked required/optional, with current values.
- **FR-2.** Editing MUST render the correct input per field type and validate required
  fields before save; "prefer not to say" MUST be honored where the schema allows.
- **FR-3.** A **"Research studies"** section MUST list studies the viewer is eligible
  for with title, purpose, consent text, and current consent state; opting in/out MUST
  POST consent and reflect immediately; withdrawal MUST be possible at any time.
- **FR-4.** All writes MUST confirm success/failure and never silently drop input.
- **FR-5.** Sections MUST hide entirely when the org enables no custom fields and no
  studies (no empty scaffolding).

## 6. Non-Functional Requirements

- **Security** — writes authenticated via existing token store; consent changes audited
  server-side (existing).
- **Privacy & Compliance** — FERPA/IRB: demographics optional, consent is informed and
  revocable; no demographic/consent values in logs or analytics payloads.
- **Accessibility** — labeled inputs, error association, ≥44pt targets, Dynamic Type.
- **Reliability** — idempotent saves; consent toggle reflects server truth on reload.
- **Internationalization** — field labels/consent text from server respect locale.

## 7. Acceptance Criteria

- **AC-1.** *Given* org custom fields, *When* profile opens, *Then* they render with
  current values and required markers; saving persists and re-fetches confirm.
- **AC-2.** *Given* a research study, *When* the user consents, *Then* the server records
  consent and the UI shows "enrolled," and withdrawing reverses it.
- **AC-3.** *Given* a required field left blank, *When* saving, *Then* a clear inline
  error blocks save.
- **AC-4.** *Given* no custom fields and no studies, *When* profile opens, *Then* the
  sections are hidden.
- **AC-5.** Both apps build.

## 8. Data Model

- No server changes. Client renders server-provided field schemas; no local persistence
  beyond transient form state.

## 9. API Surface

- Reuse `demographics` (`GET/PUT /api/v1/me/demographics` per `demographics-api.ts`),
  `custom-fields` (`custom-fields-api.ts`), and research consent
  (`server/internal/httpserver/research_consent_http.go`, `research-consent-api.ts`).
  Verify exact paths/shapes when implementing.

## 10. UI / UX

- Profile → "Personal details" card (demographics + custom fields, edit sheet per group).
- Profile → "Research studies" card with per-study row → detail/consent sheet.
- Reuse settings form components from [M1.4](../completed/mobile/M1.4-settings-accommodations.md).
- States: empty (hidden), loading skeleton, save error banner, success toast.

  Research consent reuses existing `/api/v1/me/consent-studies` routes.
- **iOS**: `ProfilePersonalDetailsView`, `ResearchStudiesView`, `ProfileDepthCards`;
  `ProfileDepthLogic.swift`, `ProfileDepthModels.swift`, API in `LMSAPIFeatures.swift`.
- **Android**: `ProfilePersonalDetailsScreen`, `ResearchStudiesScreen`, `ProfileDepthStrings`;
  parity logic in `ProfileDepthLogic.kt`; `ProfileTab` visibility + navigation.
- **Feature flag**: `ffMobileProfileDepth` on `PlatformFeatures` with client pref
  `mobile_profile_depth` (defaults on; hides sections when disabled).
- **i18n**: `mobile.profileDepth.*` keys in all five `clients/mobile/locales/*.json` (synced).
- **Tests**: `ProfileDepthLogicTests` (iOS, passing on simulator); `ProfileDepthLogicTest` (Android);
  `me_profile_depth_nodb_test.go` (server route registration).
