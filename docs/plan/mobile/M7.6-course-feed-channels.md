## 1. Problem Statement

The course feed is a real-time, chat-style stream of channels where instructors and
students post messages, share images, like, and pin — the day-to-day pulse of a class.
It is distinct from threaded discussions (M7.1): the feed is fast, social, and live over
a websocket. Mobile users are cut out of class conversation entirely, which for a
phone-first cohort is the wrong place to be absent.

## 2. Goals

- Read and post in **course feed channels** in real time on mobile.
- Support **likes**, **pins** (staff), and **image attachments** as the web does.
- Reuse the same feed for **group channels** ([M7.4](./M7.4-groups-collab.md)).
- Live updates via the existing **feed websocket**, with offline read + queued send.
- Sensible **notifications** for mentions/new messages, deep-linking into the channel.

## 3. Non-Goals

- Not threaded discussions/forums — that's [M7.1](./M7.1-discussions.md).
- No channel administration/moderation config on mobile v1 beyond pin/delete-own
  (moderation dashboard stays web).
- No rich composer beyond text + image v1 (files/video later).

## 4. Personas & User Stories

- **As a student**, I open my course feed, read what's new live, and post a question.
- **As a student**, I like a helpful post and see pinned announcements at the top.
- **As an instructor**, I post to a channel, pin an important message, and share a photo.
- **As a group member**, I chat in my group's channel using the same feed.

## 5. Functional Requirements

- **FR-1.** The workspace **Feed** section MUST list channels
  (`GET /courses/{c}/feed/channels`) and open a channel to its messages
  (`…/channels/{id}/messages`), newest-relevant ordering with pinned messages surfaced.
- **FR-2.** Composing MUST POST a message; **image attach** MUST use
  `…/feed/upload-image`; sends MUST be optimistic with a failed/retry state; offline sends
  queue and replay (M0.2).
- **FR-3.** **Like** (`…/messages/{id}/like`) and, for staff, **pin**
  (`…/messages/{id}/pin`) and delete (`…/feed/messages/{id}`) MUST be available with
  correct role gating; the user's own messages are editable/deletable.
- **FR-4.** The channel MUST connect to the **feed websocket** (`…/feed/ws`) for live
  message/like/pin updates; on disconnect it MUST reconnect and reconcile missed messages.
- **FR-5.** **Group channels** MUST reuse the same UI via the group feed routes
  (`/courses/{c}/groups/{group_id}/feed/channels[/{id}/messages]`).
- **FR-6.** A **roster** affordance (`…/feed/roster`) MUST resolve author identities and
  power @mention rendering.
- **FR-7.** New messages/mentions MUST raise notifications (M0.1) that deep-link to the
  channel at the message.

## 6. Non-Functional Requirements

- **Performance** — channel open < 500 ms from cache; live message render < 200 ms;
  image upload shows progress.
- **Security** — websocket authenticated with the existing token; role-gated pin/delete
  re-checked server-side; uploaded images scanned per existing AV pipeline.
- **Privacy & Compliance** — FERPA: feed membership scoped to enrolled/authorized;
  no cross-course leakage; report/abuse path links to existing moderation.
- **Accessibility** — message list navigable, ≥44pt actions, live updates announced
  politely (not spammed), images require alt where the composer supports it.
- **Reliability** — websocket auto-reconnect with backoff; gap-fill on reconnect;
  idempotent send (client message id) to avoid dupes.
- **Internationalization** — timestamps localized; RTL message layout.
- **Offline** — last messages cached and readable; sends queue with a clear pending state.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course feed, *When* opened, *Then* channels list and a channel shows
  messages with pinned ones surfaced.
- **AC-2.** *Given* two devices in a channel, *When* one posts/likes/pins, *Then* the other
  reflects it live via the websocket within the latency budget.
- **AC-3.** *Given* a composed message with an image, *When* sent, *Then* it uploads,
  appears optimistically, and persists; a failed send shows retry.
- **AC-4.** *Given* a non-staff user, *Then* pin/delete-others controls are absent; staff
  see them and they work.
- **AC-5.** *Given* offline, *When* posting, *Then* the message queues and sends on
  reconnect without duplication.
- **AC-6.** *Given* a group, *Then* its channel uses the same UI via group feed routes.
- **AC-7.** Both apps build.

## 8. Data Model

- No server schema changes. Client caches channels + recent messages per channel; an
  outbox table for queued sends keyed by client message id (M0.2).

## 9. API Surface

- Reuse `GET /courses/{c}/feed/channels`, `GET/POST …/channels/{id}/messages`,
  `POST …/messages/{id}/like`, `POST …/messages/{id}/pin`,
  `DELETE/PATCH …/feed/messages/{id}`, `POST …/feed/upload-image`, `GET …/feed/roster`,
  WS `…/feed/ws`, and the group-feed equivalents. Verify shapes vs `course-feed-api.ts`,
  `group-spaces-api.ts`, and `feed_websocket.go`.

## 10. UI / UX

- Workspace **Feed** chip → channel list → channel (chat-style: messages, pinned banner,
  like counts, author avatars from roster).
- Composer: text + image button + send; optimistic bubble with pending/failed state.
- Long-press message → like/pin/edit/delete/report per role.
- States: empty channel, loading, reconnecting banner, offline (cached + pending), error.
- Reuse Inbox message-bubble styling where it fits; image preview via Files viewer.

## 11. AI / ML Considerations

None in v1 (moderation/AI stays server-side).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Duplicate/lost messages on reconnect | M | H | Client message id + gap-fill on reconnect |
| WS battery/network cost | M | M | Connect only on active channel; backoff; foreground-only |
| Image upload abuse / unscanned content | L | M | Reuse existing AV scan + size/type limits |

## 15. Rollout Plan

- Flag: `mobile_course_feed`. Internal → pilot → GA.
- Sequence: read channels → live updates → compose/image → like/pin → group channels.
- Rollback: flag hides the Feed section.

## 16. Test Plan

- **E2E** — two-client live updates; offline send replay; mention push → deep link.
- **Security** — staff-only pin/delete; WS auth; scanned uploads.
