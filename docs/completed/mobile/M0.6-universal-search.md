## 1. Problem Statement

The web app ships a global search and a command-palette "go to" that lets users jump to
any course, module item, person, or action in two keystrokes. The mobile app has none —
users must remember which tab and how many taps a thing lives behind. As the app grows
to dozens of surfaces (M0.5), browse-only navigation becomes the primary friction.

## 2. Goals

- One search affordance reachable from every root tab that finds **courses, content
  (modules/pages/assignments/quizzes), people, and in-app actions**.
- Fast, forgiving, type-ahead results that deep-link straight to the destination.
- A lightweight **command-palette** mode for jump-to-action ("Take attendance",
  "My grades", "Open Calendar") mirroring the web `command-palette-go-to`.
- Recent searches + recent destinations for one-tap return.

## 3. Non-Goals

- No new search backend or ranking work — consume existing endpoints as-is.
- Not building admin search (`/admin/search/*` stays web/admin).
- Not full-text search inside file contents beyond what the API returns.

## 4. Personas & User Stories

- **As a student**, I type "midterm" and jump straight to the quiz, no matter which
  course it's in.
- **As an instructor**, I type a student's name and open their profile/progress.
- **As any user**, I type "calendar" and the palette offers "Open Calendar" as an action.
- **As a returning user**, my recent searches and destinations are one tap away.

## 5. Functional Requirements

- **FR-1.** A search entry (header icon, from M0.5) MUST open a full-screen search
  surface with an autofocused field and keyboard up.
- **FR-2.** Querying MUST call the universal search endpoint(s) with debounce
  (≥250 ms), show typed result groups (Courses · Content · People · Actions), and render
  empty/loading/error/offline states.
- **FR-3.** Selecting a result MUST deep-link to its destination via the M0.5 router
  (course → workspace, item → item detail, person → profile, action → that screen).
- **FR-4.** A **command/action** mode MUST surface in-app actions (registry from M0.5
  destinations) matched by label/synonyms, executable directly.
- **FR-5.** The surface MUST persist **recent searches** and **recent destinations**
  locally and offer them when the query is empty.
- **FR-6.** Search MUST respect authorization — results MUST only include what the
  server returns for the viewer (no client-side widening).
- **FR-7.** Scoped search: when opened from inside a course workspace, the field MUST
  offer a "in this course" toggle that narrows to that course's content.

## 6. Non-Functional Requirements

- **Performance** — first results < 400 ms p95 on warm network; debounced to avoid
  request storms; cancel in-flight on new keystroke.
- **Security** — server-authoritative results; no PII cached beyond display fields;
  recent searches stored in app-local store only.
- **Privacy** — recent searches clearable from Settings; no logging of query text to
  analytics beyond counts.
- **Accessibility** — results list is screen-reader navigable with group headers as
  headings; ≥44pt rows; result count announced.
- **Reliability** — offline shows recents + a clear offline notice; partial-endpoint
  failure degrades the affected group only.
- **Internationalization** — query and results respect locale; RTL layout.

## 7. Acceptance Criteria

- **AC-1.** *Given* any root tab, *When* the user opens search and types, *Then* grouped
  type-ahead results appear within the latency budget and update as they type.
- **AC-2.** *Given* a content result, *When* selected, *Then* the app deep-links to that
  exact item with a correct back stack.
- **AC-3.** *Given* the query "calendar", *When* shown, *Then* an "Open Calendar" action
  appears and launches the calendar.
- **AC-4.** *Given* an empty query, *When* the surface opens, *Then* recent searches and
  recent destinations are shown and tappable.
- **AC-5.** *Given* offline, *When* searching, *Then* recents show with an offline notice
  and no crash.
- **AC-6.** Both apps build.

## 8. Data Model

- No server changes. Client-only: `recentSearches` and `recentDestinations` lists in the
  app prefs/cache store, capped and clearable.

## 9. API Surface

- Reuse `GET /api/v1/search` / `GET /api/v1/search/query` (primary universal search) and,
  where relevant per scope, `/library/search`, `/oer/search`, `/standards/search`,
  `/concepts/search`. Confirm exact request params + response shape against
  `server/internal/httpserver/search.go` and `clients/web/src/lib/search-api.ts` when
  implementing. Rate-limit handled by existing API client; debounce client-side.

## 10. UI / UX

- Full-screen modal/route: search field + cancel, grouped results with section headers
  and per-row icon/title/subtitle (course code, due date, role).
- Action/command rows visually distinct (leading action glyph).
- Empty state = recents; no-results state = "No matches for '…'" + suggestions.
- Reuse existing list/row components and `LMSEmptyState`/error banner.
- Scoped pill ("In this course") when launched from a workspace.

## 11. AI / ML Considerations

None in v1 (server may already rank; mobile just renders). Future: semantic "ask" could
hand off to [M7.2](./M7.2-ai-tutor.md).

## 12. Integration Points

- **iOS:** new `Features/Search/UniversalSearchView.swift`, search calls in
  `Core/LMS/LMSAPIFeatures.swift`, models in `Core/LMS/LMSFeatureModels.swift`; header
  hook in `Features/Home/MainTabView.swift` (M0.5). Regenerate xcodeproj.
- **Android:** new `features/search/UniversalSearchScreen.kt`, calls in
  `core/lms/LmsApi.kt`, models in `core/lms/LmsFeatureModels.kt`; header hook in
  `features/home/HomeScreen.kt`.
- Routes through the M0.5 deep-link router / destination registry.

## 13. Dependencies & Sequencing

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Endpoint param/shape drift | M | M | Verify against server + web api lib at build time; thin adapter |
| Request storms on fast typing | M | M | Debounce + cancel in-flight |
| Over-broad results / authz leak | L | H | Server-authoritative only; no client widening |

## 15. Rollout Plan

- Flag: `mobile_universal_search`. Internal → pilot → GA.
- Ship behind the M0.5 header icon (dark until enabled).
- Rollback: flag off hides the entry point.

## 16. Test Plan

- **Unit** — debounce/cancel; recents capping; action matcher.
- **Integration** — endpoint adapter shapes; scoped vs global.
- **E2E** — type → grouped results → deep-link each type; action launch; offline recents.
- **Accessibility** — grouped headings, announcements, target sizes.
