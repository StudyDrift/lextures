## 1. Problem Statement

Instructors run live classes, office sessions, and review meetings through Lextures'
virtual-meetings layer (provider-backed: Zoom/Meet/etc, plus an in-app whiteboard).
A mobile student can't see the schedule, can't tap to join, and isn't counted in meeting
attendance — so the phone-first student misses class entirely. This is one of the
largest remaining parity gaps.

## 2. Goals

- See **upcoming/live/past** meetings for a course (and across courses on Home/Calendar),
  with clear "Live now" and "Starts in N" states.
- **Join** a meeting from the app via the provider's mobile path (deep link/SDK/web view).
- Be counted in **meeting attendance** when joining where the meeting tracks it.
- View an **in-app whiteboard** tied to a session (read/participate per the board's mode).
- Let an **instructor** start/end and see who's present (lightweight controls; full
  scheduling stays web).

## 3. Non-Goals

- No bespoke in-app video stack — defer real-time A/V to the configured provider.
- No meeting scheduling/config authoring on mobile v1 (start/end + join only).
- No recording management (web/admin).

## 4. Personas & User Stories

- **As a student**, I see today's class is live and tap "Join" to enter.
- **As a student**, joining marks me present without a separate check-in.
- **As an instructor**, I start the session and see the live roster from my phone.
- **As a student**, I open the session whiteboard to follow along.

## 5. Functional Requirements

- **FR-1.** A course workspace **Live** section MUST list meetings
  (`GET /courses/{c}/meetings`) with status (scheduled/live/ended/cancelled), time, and
  provider, plus a cross-course "Live & upcoming" surface on Home/Calendar.
- **FR-2.** A meeting detail MUST show title, time, provider, join affordance, and (staff)
  start/end; **Join** MUST use the provider join info (`getMeetingJoinInfo`) opening the
  provider app/deep link or an in-app web view fallback.
- **FR-3.** Joining MUST record **attendance** where the meeting supports it
  (`/meetings/{id}/attendance`); the user's present state MUST reflect back in the UI.
- **FR-4.** A meeting with an associated **whiteboard** MUST offer opening it
  (`/courses/{c}/whiteboards/{board_id}`) in a native viewer; v1 may be read/follow with
  participate as feasible.
- **FR-5.** "**Add to calendar**" MUST be available via the meeting ical
  (`/meetings/{id}/ical`) and meetings MUST appear in [M2.1](../completed/mobile/M2.1-calendar-todos.md).
- **FR-6.** Instructors MAY **start/end** a meeting and see the live attendee list;
  scheduling/config remains web with a "manage on web" affordance.
- **FR-7.** Live state MUST update (poll or reuse a realtime channel) so "Live now"
  appears promptly and "Join" enables at start.

## 6. Non-Functional Requirements

- **Performance** — meeting list < 500 ms p95; "Live now" reflects within ~30 s of start.
- **Security** — join info fetched per-request, not cached; provider tokens never logged;
  attendance authenticated; staff controls re-check role server-side.
- **Privacy & Compliance** — FERPA: rosters/attendance visible only to authorized staff;
  no recording exposed beyond provider policy.
- **Accessibility** — join/whiteboard controls labeled, ≥44pt; live-status announced;
  whiteboard provides a non-visual fallback/notice when content is purely visual.
- **Reliability** — provider unreachable → clear error + retry; attendance write queued
  if it fails at join.
- **Internationalization** — times in user tz/locale; RTL.

## 7. Acceptance Criteria

- **AC-1.** *Given* a course with meetings, *When* the student opens Live, *Then* meetings
  list with correct status and a live one shows "Join."
- **AC-2.** *Given* a live meeting, *When* the student taps Join, *Then* the provider
  opens via deep link/web view and attendance records present.
- **AC-3.** *Given* a meeting with a whiteboard, *When* opened, *Then* the board renders
  in a native viewer.
- **AC-4.** *Given* an instructor, *When* they start/end a meeting, *Then* state changes
  server-side and the live roster shows; scheduling links out to web.
- **AC-5.** *Given* any meeting, *When* "Add to calendar," *Then* the ical imports and it
  appears in the in-app calendar.
- **AC-6.** Both apps build.

## 8. Data Model

- No server schema changes. Client caches meeting lists per course; queues attendance
  write on failure (M0.2). No provider credentials stored beyond the transient join token.

## 9. API Surface

- Reuse `GET /api/v1/courses/{c}/meetings`, `GET/PATCH /api/v1/meetings/{id}`,
  meeting join-info, `POST /api/v1/meetings/{id}/attendance`, `GET …/ical`,
  `GET/POST /api/v1/courses/{c}/whiteboards[/{board_id}]`, provider list
  (`virtualmeetings_providers`). Verify join-info + attendance shapes vs
  `virtual-meetings-api.ts` and `virtualmeetings_*.go`.

## 10. UI / UX

- Workspace **Live** chip → meeting list (status chips: Live/Scheduled/Ended) → detail.
- Home/Calendar "Live & upcoming" rail (coral "Live now" accent).
- Join = primary CTA; whiteboard = secondary; "Add to calendar"; staff start/end + roster.
- Whiteboard native viewer (pan/zoom; participate as feasible) with "open on web for full
  tools" affordance.
- States: none/empty, loading, provider-error, ended (read-only), offline (cached list).

## 11. AI / ML Considerations

None (unless a provider offers AI captions — defer to [M6.3](./M6.3-immersive-reader.md)).

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Provider lacks clean mobile deep link | M | H | Web-view fallback; document per-provider behavior |
| Whiteboard real-time too heavy for native v1 | H | M | Read/follow v1; "open on web" for authoring/heavy edit |
| Attendance not recorded on provider-side join | M | M | Record via our attendance endpoint at Join; reconcile |
| App Store rules around external video | L | M | Use provider SDK/web view per platform guidelines |

## 15. Rollout Plan

- Flag: `mobile_live_meetings`. Internal → pilot (one provider) → GA (all providers).
- Sequence: list/detail/join → attendance → whiteboard view → staff start/end.
- Rollback: flag hides Live + rails.

## 16. Test Plan

- **Unit** — status derivation; join-info handling; attendance queue.
- **Integration** — list/detail/join per provider; attendance write; ical import.
- **E2E** — student joins live → marked present; instructor start/end → roster; whiteboard
  open.
- **Security** — staff controls role-gated; join tokens not logged.
