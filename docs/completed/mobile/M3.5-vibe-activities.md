## 1. Problem Statement

"Vibe activities" are AI-generated interactive learning activities that appear as module
items on the web (`course-module-vibe-activity-page`, `fetchModuleVibeActivityByItem`).
On mobile they have no renderer, so a student tapping one in a module gets nothing or a
generic fallback. Until mobile renders them, modules that use vibe activities are
incomplete on the phone, breaking the "do a full week of coursework without a browser"
bar.

## 2. Goals

- Render a vibe activity item natively from its existing payload so students can read and
  interact with it on a phone.
- Record progress/last-visited the same way other module items do.
- Degrade gracefully for any activity sub-type mobile can't fully render, with an honest
  "best on a larger screen / open on web" path.

## 3. Non-Goals

- **Authoring/generation** of vibe activities (the `/vibe-activities/generate` instructor
  flow) stays web-only.
- No new interactive runtime beyond what the payload + existing renderers support.

## 4. Personas & User Stories

- **As a student**, I open a vibe activity in a module and complete it on my phone like
  any other content.
- **As a student**, if an activity needs a bigger screen, I get a clear notice and a way
  to open it on web — never a blank screen.
- **As an instructor**, I trust that assigning a vibe activity won't strand mobile users.

## 5. Functional Requirements

- **FR-1.** A `vibe_activity` module item MUST open a native viewer fed by
  `GET /courses/{c}/vibe-activities/{item_id}` (mirroring `fetchModuleVibeActivityByItem`).
- **FR-2.** The viewer MUST render the activity's content (prompt/body, any structured
  blocks) using the existing markdown/LaTeX renderer (`NotebookMarkdown.*`) and supported
  interaction primitives.
- **FR-3.** Interactions the payload defines (e.g. reveal, check, free-response) MUST be
  usable; where a sub-type isn't supported, the viewer MUST show a graceful "open on web"
  card without blocking the rest of the activity or module.
- **FR-4.** Opening the activity MUST record last-visited / progress consistent with
  other module items (conditional-release counters, [M3.4](../completed/mobile/M3.4-conditional-release.md)).
- **FR-5.** The viewer MUST honor conditional-release gating: a locked vibe activity
  shows the standard locked state, not its content.

## 6. Non-Functional Requirements

- **Performance** — first render < 500 ms from cached payload; no layout jump on math.
- **Security** — content fetched with the viewer's auth; no client-side unlock of gated
  items.
- **Accessibility** — interactive elements labeled, ≥44pt, Dynamic Type, VoiceOver order.
- **Reliability** — partial/unknown blocks render a placeholder, never crash.
- **Internationalization** — content respects locale; RTL layout.
- **Offline** — last-cached activity readable with a staleness indicator (M0.2).

## 7. Acceptance Criteria

- **AC-1.** *Given* a module with a vibe activity, *When* tapped, *Then* its content
  renders natively (text + math) and supported interactions work.
- **AC-2.** *Given* an unsupported activity sub-type, *When* opened, *Then* a clear
  "open on web" card shows and the rest of the activity/module remains usable.
- **AC-3.** *Given* conditional release locking the item, *When* tapped, *Then* the
  locked state shows and content is not fetched/revealed.
- **AC-4.** *Given* opening the activity, *Then* last-visited/progress updates like other
  module items.
- **AC-5.** Both apps build.

## 8. Data Model

- No server changes. Reuse the existing `ModuleVibeActivityPayload` shape on the client;
  cache last activity per item for offline.

## 9. API Surface

- Reuse `GET /api/v1/courses/{course_code}/vibe-activities/{item_id}` and module structure
  listing `…/structure/modules/{module_id}/vibe-activities`. Confirm response shape vs
  `courses-api.ts` `fetchModuleVibeActivityByItem`. No writes in v1 unless the payload
  defines a server-recorded interaction (then reuse that route).

## 10. UI / UX

- New viewer reached from module item routing (extend `ItemDetailView` /
  `ItemDetailScreen` `vibe_activity` branch).
- Header (title), scrollable body (markdown/LaTeX), interaction blocks inline.
- States: locked (conditional release), loading skeleton, error banner, unsupported-block
  card, offline staleness chip.

## 11. AI / ML Considerations

- Activities are AI-generated server-side; mobile renders the produced payload only. No
  on-device model, no prompt handling, no PII sent. If the activity defines an AI-backed
  interaction (e.g. an inline check), it routes through the existing server endpoint, not
  a new client model call.

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Activity payload richer than mobile renders | H | M | Render supported blocks; graceful "open on web" for the rest |
| Interaction recording route unclear | M | M | v1 read/interact-locally; defer server-recorded interactions until shape confirmed |

## 15. Rollout Plan

- Flag: `mobile_vibe_activities`. Internal → pilot → GA.
- Rollback: flag → fall back to current "open on web" notice for the item.

## 16. Test Plan

- **Unit** — block renderer switch; unsupported-block fallback.
- **Integration** — payload fetch + render; locked-item gating.
- **E2E** — open activity → interact → progress recorded; unsupported sub-type path.
- **Accessibility** — interaction labels, Dynamic Type.

## 19. Implementation notes

- **Routing**: `vibe_activity` now uses `ModuleItemDestination.VibeActivity` / `.vibeActivity` instead of the M3.3 web container.
- **iOS** (`VibeActivityLogic.swift`, `VibeActivityView.swift`, `ModuleLastVisited.swift`):
  - Parses instructor HTML into headings, paragraphs, lists, `<details>` reveals, button reveals, check buttons, and free-response fields.
  - Renders text/math via `CourseMarkdownContentView`; unsupported blocks (iframe/canvas/video) show an honest “open on web” card.
  - Offline cache key `vibe-activity`; last-visited stored locally; `onProgressChanged` after load.
- **Android** (`VibeActivityLogic.kt`, `VibeActivityScreen.kt`, `ModuleLastVisited.kt`): parity with iOS.
- **Feature flag**: `ffMobileVibeActivities` on `PlatformFeatures` (defaults on; set `false` to show web-only fallback).
- **Tests**: `VibeActivityLogicTests` / `VibeActivityLogicTest`; routing tests updated in `ModuleContentModelsTests`.
